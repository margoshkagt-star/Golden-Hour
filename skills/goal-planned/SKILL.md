# Goal Planned — goal-planned

**Use when:** пользователь приходит с целью (учебной / проектной / олимпиадной / личной) и хочет превратить её в действующий план. Или приносит сразу несколько целей — тогда распределяем между ними общий бюджет времени.

**Don't use when:**
- Цель тривиальная, формальный план не нужен
- План уже есть, нужно только напоминание → `cron`
- Нужно разобрать кашу в голове по текущим мелким задачам → `evening-task-triage`
- Нужен активный список задач на сегодня → `current-tasks`

## Назначение

Универсальный планировщик целей с двумя режимами:
- **Режим планирования** — сбор входных, research, 3 подробных черновика, выбор.
- **Режим сопровождения** — ежедневное составление плана на день N (с учётом результатов дня N-1), выдача материалов по запросу, мягкий контроль фокуса.

**Не отключается сам** после выбора плана. Продолжает работать, пока цель активна (до явного «закрываем» / «план завершён» / отмены).

**Не привязан к конкретному пользователю:** никаких hardcoded лимитов или предпочтений. Все параметры запрашиваются.

## Принципы

### 1. Подробность черновиков

Каждый черновик — это не «идея в одном абзаце», а почти готовый мини-план: фазы с длительностью, разбивка по неделям/дням, объём материала, ресурсы, milestone'ы, формат самопроверки, типичные ловушки, план Б. Три черновика — три разных «мира», каждый из которых пользователь мог бы взять и начать исполнять уже сегодня.

### 2. Удержание на теме (контроль дрейфа)

При работе с целью (планирование, чек-ины, адаптации) **агент активно следит за фокусом** и мягко возвращает к теме, если пользователь уезжает:

- В чек-ине пользователь перечисляет 10 задач, из них 1 — по цели, 9 — побочное → сначала зафиксировать прогресс по цели, потом уточнить, что с остальным.
- В `evening-task-triage` появляются задачи, конкурирующие с бюджетом цели → пометить конфликт, не дать сьехать в «аа, потом разберусь».
- На адаптации пользователь говорит «давай переключимся на X» → спросить «это новая цель или относится к текущей?», при необходимости запустить `goal-planned` заново.
- Жёсткого запрета нет — это не тюрьма. Но по умолчанию фокус на текущей цели до явного «закрываем».

**Как сигналить пользователю:** короткая реплика вроде «Замечу — это в сторону от цели по физике. Сначала закроем milestone X, потом разберёмся?». Без нравоучений, без морализаторства.

### 3. Агент не отключается сам

После выбора плана и записи в файл навык **не уходит в «фон»**. На каждой итерации (новый день / запрос «что делать сегодня?» / отчёт о вчерашнем) он:

- Подхватывает контекст цели (читает файл плана).
- Учитывает факт/план по предыдущим дням.
- Выдаёт план на текущий день N в формате дня 1.
- Готов к адаптации, если вчера не успели.

«Тишина» между днями ≠ отключение. Следующий контакт по цели = следующая итерация.

### 4. Материалы — по запросу, не в плане

План дня — это **что делать** (задачи по часам, конкретные действия, ожидаемый результат), а **не учебник**. Никаких видео, ссылок, разборов, текстов задач, шпаргалок внутри таблицы плана. Иначе он распухает и перестаёт быть рабочим инструментом.

**Правило для агента:**
- При **первой выдаче плана** (день 1) и при **начале новой сессии по цели** — явно сказать пользователю: «материалы выдам по запросу — скажи "дай материал для задачи X" / "покажи разбор" / "ссылку"».
- Дать **мини-меню типовых запросов** (чтобы юзер в стрессе не мучился формулировкой): «разбери задачу X» / «дай теорию по теме Y» / «покажи пример решения» / «найди видео по Z».
- **Не выдавать** материалы проактивно. Только по явной просьбе.
- Если пользователь сам просит «включи всё в план» — окей, включить, но это исключение, а не правило.

**Что может быть в плане (без материалов):**
- «Решить 3 задачи на закон Кулона» ✓
- «Прочитать §2 Мякишева, выписать формулы» ✓
- «Посмотреть 1 видео по теме, выбрать лучшее» — упоминание видео как действия, но без ссылки

**Что НЕ должно быть в плане (материалы):**
- ❌ Сама ссылка на видео
- ❌ Текст задачи
- ❌ Полный разбор
- ❌ Готовые формулы в шпаргалке (это артефакт работы, а не план)

**Граница «источник vs контент»:**
- В плане ОК: **указатели на источник** — «§2 Мякишева», «глава 4», «раздел про ёмкость», «любое видео по теме X на YouTube». Это ссылки на внешнее, не сам контент.
- В плане НЕ ОК: **контент** — конкретная ссылка, текст задачи, готовый разбор, заполненная шпаргалка.
- В файле плана в секции `Research-сводка` — ОК любые ссылки и цитаты (это архив, не рабочий план).

## Входные данные

### Глобальный параметр (запрашивается один раз в начале)

- **Бюджет времени** — сколько часов в день (или в неделю) пользователь готов тратить на ВСЕ цели суммарно. Не задаётся заранее, спрашиваем.
- **Единицы бюджета:** если юзер назвал ч/неделю — нормализовать в ч/день одним из способов:
  - **Поровну:** ч/неделю ÷ 7.
  - **Рабочие дни:** ч/неделю ÷ 5 (Пн–Пт плотно, выходные 0).
  - **Гибкое:** «5 дней × N ч, 2 дня × 0» или похоже.
  - Уточнить у юзера, какой режим, и зафиксировать.

### На каждую цель (4 поля + язык)

| Поле | Что значит | Формат |
|---|---|---|
| **Вес** | Приоритет; при нескольких целях время распределяется пропорционально | Число 1–10 или low/med/high. **Валидация:** должен быть > 0. Если 0 или отрицательный — спросить, не выдумывать. |
| **Дедлайн** | Контрольная дата | Дата. **Валидация:** не в прошлом. Если в прошлом — спросить: сдвинуть или закрыть цель. |
| **Сложность** | Текущий уровень пользователя по теме | Свободный ввод: что знаю/не знаю, опыт |
| **Категория** | Тип цели | Строка: физика, программирование, язык, олимпиада… |
| **Язык** | Язык материалов и формулировок | ru / en / mixed. Использовать для research и для текста в плане дня. |

### Бюджет = 0 (edge case)

Если юзер говорит «у меня 0 ч/день на эту цель» — не строить план. Ответить одной строкой: «с 0 ч/день план сделать нельзя. Хочешь минимальный 30 мин/день, или это ошибка ввода, или пока зафиксировать цель как `на паузе`?»

### По материалам (уточняется по каждой цели)

- **Свои** — учебники, конспекты, лекции, что уже есть
- **Внешние** — подобрать через research, если попросят

## Процедура

### 1. Сбор входных

- Спросить глобальный бюджет времени (если не дан). Если в ч/неделю — уточнить режим распределения (поровну / рабочие дни / гибкое).
- На каждую цель — 4 поля + язык. Если не даны — спросить по одному, максимум 1–2 уточнения за раз.
- Materials: спросить, что у пользователя уже есть по каждой цели.
- Валидация (см. «Входные данные»): вес > 0, дедлайн не в прошлом, бюджет > 0.

### 2. Research (по каждой цели)

- `web_search` + `web_fetch` по открытым источникам: curricula, roadmaps, syllabus, формат олимпиад/экзаменов, разборы задач, типичные ошибки.
- **Процедура факт-чека** (не «на глаз», а конкретно):
  1. Дата публикации источника < 2 лет. Если старше — пометить `[требует проверки]`.
  2. Подтверждение из 2+ независимых источников. Если один — пометить `[единичный источник]`.
  3. Для форматов экзаменов / олимпиад / квалификаций — обязательно официальный сайт (ФИПИ, вуз, оргкомитет).
  4. Итог в файле: либо ссылка с датой и оценкой надёжности, либо пометка «не подтверждено».
- Сохранить ключевые ссылки и сводку в файл плана (секция `Research-сводка` — можно с контентом и ссылками).
- **Не выдумывать:** если по теме нет публичных материалов — сказать честно, поискать смежные области.
- **Если инструменты research недоступны** (offline, API down) — сообщить и продолжить без research, пометив в файле плана `research: не выполнен (<причина>)`.

### 3. Подробные черновики (≥2, лучше 3)

Обязательно с альтернативами: разные подходы, темп, scope. Каждый черновик содержит следующие блоки:

- **Суть подхода (1–2 предложения):** как именно идём к цели, в чём ключевая идея.
- **Фазы:** таблица — номер, название, что внутри, длительность.
- **Разбивка по неделям/дням:** что делается, какие занятия, в каком порядке.
- **Конкретные темы / модули:** список того, что покрывается (темы, главы, функции, навыки).
- **Объём материала:** цифры — страницы, главы, часы видео, кол-во задач, километры, повторения.
- **Ресурсы по фазам:** какие книги, сайты, видео, инструменты открываем на каждой фазе.
- **Milestone'ы (измеримые):** чекбоксы с датами.
- **Критерии готовности фазы:** как пользователь сам поймёт, что фаза закрыта.
- **Формат самопроверки:** как проверяем прогресс.
- **Типичные ловушки и противодействие:** что обычно мешает + как обходить.
- **Дневная нагрузка:** ... ч. **Примечание:** если нагрузка меняется по фазам — указать диапазон или таблицу по фазам.
- **Плюсы / минусы:** trade-off'ы.
- **План Б при отставании:** что делать, если отстали на N дней (конкретные шаги).

**Самопроверка черновика перед выдачей:** все 14 блоков заполнены? Если «…» / «TBD» — дозаполнить, не выдавать заглушку.

### 4. Расчёт дневной нагрузки

- Бюджет делится между целями пропорционально весам.
- На каждую цель: `(бюджет_в_день × вес_цели / сумма_весов)`.
- Если цель одна — весь бюджет ей.
- **Валидация:** если сумма весов = 0 — спросить юзера (см. «Входные данные»).
- Показать формулу и логику, не магическое число.

### 5. Capacity check

- Если требуемая нагрузка > бюджета — флажок «не влезет при заданном бюджете».
- Варианты реакции: продлить таймлайн / увеличить бюджет / урезать scope.
- **При сильном недовлезании** (план требует >1.5× бюджета) — обязательно пометить, что план рискованный, и предложить урезание.

**Edge case: дедлайн слишком близок (triage-режим).**
- **Если до дедлайна <24 ч** — это не план, а triage. Вместо полной процедуры (research → 3 черновика → выбор) агент:
  1. Спрашивает: «Дедлайн через <X ч. Полный план сделать не успеем. Хочешь triage — 3 самых важных действия на оставшееся время, без черновиков и фаз?»
  2. При согласии — выдаёт 3 действия с конкретным временем и сразу план на оставшиеся часы.
  3. При несогласии — план строится как обычно, но с пометкой «цель вряд ли достижима в срок».
- **Если дедлайн в прошлом** — сказать честно: «Дедлайн уже прошёл. Хочешь сдвинуть на новую дату или закрыть цель?»

### 6. Запись плана

- Путь: `~/.openclaw/workspace/plans/<категория>-<слаг>.md` — один файл на цель.
- **Перед записью:** убедиться, что папка `plans/` существует (`mkdir -p` если нет). На Windows путь = `%USERPROFILE%\.openclaw\workspace\plans\`.
- **Слаг:** транслит категории + 3-5 слов из названия цели, латиницей или кириллицей, без спецсимволов и пробелов (заменять на `-`). Пример: `plans/ucheba-fizika-elektrodinamika-nedelya.md`. **Если файл с таким слагом уже есть** — спросить: «это та же цель (продолжаем) или новая (тогда другое имя)?»
- Содержимое: входные данные, research-сводку, черновые планы, выбранный, дневную нагрузку, историю адаптаций, журнал планов по дням, drift log.
- **После записи:** проверить, что файл существует. Если ошибка FS (прав нет, диск заполнен, путь недопустим) — сообщить юзеру и попросить помощь, **не утверждать, что план сохранён**, пока не проверено.
- Несколько целей = несколько файлов.
- Не плодить дубли при обновлении: править существующий файл, а не создавать новый.

### 7. Адаптация при срыве дедлайна

- **Триггер:** пользователь говорит «сорвал дедлайн X» / «не успеваю» / «план не работает», ИЛИ отставание по плану перешло в категорию «системное» (>3 дней, см. шаг 10).
- **Действия:**
  1. Прочитать файл плана + журнал дней.
  2. Сравнить факт/план (что сделано, что нет).
  3. Предложить 2–3 варианта коррекции:
     - Продлить таймлайн
     - Урезать scope (какие milestone'ы можно отложить)
     - Увеличить бюджет
     - Переставить порядок фаз
  4. Если план нереалистичен в текущих рамках — сказать честно, не обещать чуда.
  5. **После любой адаптации (даже перестановки фаз) — явно обновить в файле:**
     - активный milestone (если сдвинулся);
     - дневную нагрузку (если новая фаза требует больше/меньше);
     - даты milestone'ов (если переставили порядок или сдвинули дедлайн).
  6. Записать новую версию в файл плана с датой адаптации.

**Связь с шагом 9:** после шага 7 следующая выдача плана дня N идёт уже по обновлённой фазе/milestone'ам. Агент обязан перечитать файл и не полагаться на то, что помнит «старое» состояние.

### 8. Контроль дрейфа (на каждой сессии по цели)

- В начале сессии / при чек-ине: «Сначала по цели — что сдвинулось, потом остальное».
- Если пользователь приносит задачи, явно не относящиеся к текущей цели — короткое уточнение: «Это новая цель, дополнительная задача в текущей, или побочное?».
- Если новая цель — **немедленно пересчитать дневную нагрузку текущей цели** (по формуле шага 4 с учётом нового веса). Возможно, потребуется адаптация (шаг 7) текущей цели. Затем запустить `goal-planned` для новой цели.
- Если цели конфликтуют (например, «интенсивный французский за 2 нед» + «начать испанский») — пометить конфликт явно, спросить, какая приоритетнее.
- Если побочное — записать в `evening-task-triage` или отложить, не размывая бюджет текущей цели.
- **Drift log:** каждое drift-событие фиксировать в файле плана (секция `Drift log`): дата, что случилось, как среагировали. Это не «история адаптаций» — это отдельный лог дрейфа.
- Без нравоучений. Просто структура: «вот текущий фокус, вот куда ты смотришь, давай выровняем».

### 9. Ежедневный план на день N (режим сопровождения)

**Триггеры (любой из):**
- Начало нового дня по цели.
- Запрос пользователя: «что делать сегодня?» / «план на сегодня» / «давай день N».
- Отчёт пользователя о вчерашнем дне («вчера сделал только X»).
- Импульс агента: если юзер вернулся к цели впервые за >6 ч и есть открытые дни в журнале — предложить план на текущий день.

**Действия:**
1. **Определить N:** посчитать записи в `Журнал планов по дням` в файле плана. `N = кол-во_записей + 1`. Если журнал пуст → `N = 1`. Эта запись ещё не существует на момент вычисления.
2. **Определить текущую фазу:** последняя фаза, у которой (а) milestone'ы ещё не все закрыты ИЛИ (б) дедлайн фазы ещё не прошёл. Берём то, что «стоп-сигналит» позже. Если впереди — двигаемся; если позади — застреваем в текущей, не прыгаем в следующую.
3. Прочитать файл плана + журнал предыдущих дней.
4. Определить:
   - Активный milestone (первый незакрытый в текущей фазе).
   - Факт/план по предыдущим дням (что успели, что нет).
5. **Если факт/план по предыдущим дням показывает отставание — скорректировать день N под реальность (см. шаг 10).**
6. Составить **план на день N** в формате дня 1 (см. шаблон): таблица «время — что делать — конкретика» + цель дня + итог + куда идём завтра.
7. **Проверить сумму часов** в таблице: она должна быть ≤ дневной нагрузке из выбранного черновика. Если больше — сократить задачи или пометить каждую лишнюю как `[вне бюджета: +X ч]`.
8. **Не прикладывать материалы** в план (см. принцип 4). Материалы выдаются только по запросу.
9. **Обязательно сообщить правило** о материалах (при первой выдаче или начале новой сессии): «материалы выдам по запросу — скажи, если что-то нужно» + мини-меню типовых запросов.
10. **Если активных целей >1** — план на день выдаётся **единым сводным**, с разбивкой по целям (например, утро — цель 1, вечер — цель 2). Сумма часов по всем целям в один день ≤ общему бюджету.
11. Записать план в файл (секция «Журнал планов по дням → День N»). Если multi-goal — отметить цели в каждой строке таблицы (отдельная колонка «Цель»).
12. **Если юзер не отчитался за вчера** (в журнале за день N-1 «Итог дня (факт)» пуст) — одной строкой спросить: «Что вчера закрыли из плана? Или сказать `пропустить`, чтобы двигаться дальше?»

### 10. Адаптация дневного плана

- Если пользователь сообщает «вчера сделал только X из Y» или «сегодня не успеваю N часов»:
  1. Прочитать журнал дней.
  2. Классифицировать отставание:
     - **Лёгкое (1 день):** перенести часть задач на завтра, общий дедлайн не двигать.
     - **Среднее (2–3 дня подряд):** урезать scope — отбросить низкоприоритетные задачи из активной фазы, общий дедлайн сдвинуть на 1 неделю.
     - **Системное (>3 дней подряд):** пересмотреть план целиком, продлить общий дедлайн, при необходимости запустить адаптацию (шаг 7).
  3. Зафиксировать в истории адаптаций.
  4. Сразу выдать скорректированный план на текущий день (если применимо) или на следующий.

**Связь с шагом 7:** «Системное» отставание из шага 10 автоматически запускает шаг 7 (адаптация при срыве дедлайна).

### 11. Долгое отсутствие юзера

- Если юзер не возвращался к цели **>7 дней**:
  - Не авто-паузить (решает юзер).
  - При следующем контакте одной строкой: «Прошло N дней, план в режиме ожидания. Возобновляем как есть / сдвигаем дедлайн / закрываем?»

## Правила переходов статусов

Файл плана имеет поле `Статус: активен / на паузе / завершён / отменён`. Правила переходов:

| Из → В | Триггер | Что делать с файлом |
|---|---|---|
| активен → завершён | Все milestone'ы закрыты И юзер подтвердил «готово» ИЛИ (дедлайн прошёл + 100% milestone'ов) | Файл остаётся в `plans/`, статус обновляется, история финализируется |
| активен → завершён (с провалом) | Юзер сказал «сделал, что мог» при частичных milestone'ах | Файл остаётся, статус «завершён», в истории — пометка «частично: X из Y milestone'ов» |
| активен → на паузе | Юзер сказал «поставь на паузу» | Дедлайны замораживаются, не сдвигаются. В истории — дата паузы. |
| на паузе → активен | Юзер сказал «продолжаем» | Дедлайны размораживаются. Если пауза была >14 дней — предложить ревизию плана. |
| активен/на паузе → отменён | Юзер сказал «закрываем» / «забили» | Файл **не удаляется**, помечается `cancelled`, остаётся в `plans/` для истории. |

## Анти-паттерны

- ❌ Один план без альтернатив
- ❌ План без research (на глаз)
- ❌ Поверхностные черновики без конкретных занятий по дням
- ❌ Магическое «делай столько-то часов» без логики расчёта
- ❌ Допрос 10+ вопросами при сборе входных
- ❌ Забыть про уже идущие планы и перегрузить пользователя
- ❌ Нечестно обещать нереальное («успеешь, если постараешься»)
- ❌ Игнорировать materials, которые у пользователя уже есть
- ❌ Плодить копии файлов при обновлении
- ❌ «Материнский тон»: «давай, ты сможешь!», «не сдавайся!»
- ❌ Привязка к конкретному пользователю (hardcoded предпочтения, лимиты)
- ❌ Дать пользователю незаметно сьехать с темы на побочные задачи, не обозначив это
- ❌ Запрещать дрейф жёстко (это не тюрьма) — но и не замечать его
- ❌ Самоотключаться после выбора плана (навык остаётся активным до явного «закрываем»)
- ❌ Вкладывать материалы в план дня (видео, ссылки, тексты задач) — это забивает план и ломает его как рабочий инструмент
- ❌ Выдавать материалы проактивно, без запроса
- ❌ Не сообщить правило о материалах при первой выдаче плана
- ❌ Считать «день N» по наитию — только по журналу
- ❌ Выдавать план дня с суммой часов > дневной нагрузки (без пометки)
- ❌ Игнорировать «нет отчёта» за вчера — молча продолжить как будто всё ок
- ❌ Строить месячный план, когда до дедлайна <24 ч (вместо triage)
- ❌ Использовать размытые пороги «>N дней» — только конкретные 1 / 2-3 / >3
- ❌ Принимать вес ≤ 0, дедлайн в прошлом, бюджет = 0 без явного уточнения
- ❌ Говорить «план сохранён», не проверив, что файл записан
- ❌ Использовать слаг с пробелами/спецсимволами или перезаписывать существующий файл без вопроса
- ❌ Определять текущую фазу по наитию — есть правило в шаге 9
- ❌ После адаптации (шаг 7) выдавать дневной план по старой фазе
- ❌ При multi-goal выдавать 2 отдельных дневных плана (нужен единый сводный)
- ❌ Не зафиксировать drift-событие в логе
- ❌ Удалять файл плана при отмене (он остаётся в `plans/` для истории)
