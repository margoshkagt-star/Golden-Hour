# Agent Orchestration (создание и управление агентами для тяжёлых задач)

Процедура для случаев, когда задача глубокая/тяжёлая/многосоставная. Скилл формализует «когда и как» поднимать подагентов, координировать их и собирать результат.

## Когда применять

**Триггеры** (description загружается автоматически):
- «Создай агента», «подними бота», «запусти субагента»
- «Тяжёлая задача», «сложная задача», «разбери по-нормальному», «глубокий разбор»
- «Распараллель», «делегируй», «разруливай с агентами», «orchestration»
- Когда adaptive-стратегия (правило 10 в BOTS.md) указывает на 1+ подагентов или параллель

**Не применять**:
- Лёгкие задачи (adaptive → сам)
- Простой вопрос-ответ
- Уже есть подходящий cron-агент (scout, writer, ops, coder, friend, researcher)
- Карим явно сказал «сделай сам»

## Процедура

### Шаг 1. Оцени задачу (adaptive-стратегия)
- **Лёгкая** (вопрос, <30 сек) → НЕ применять этот skill
- **Средняя** (1 research, 1 отчёт) → 1 подагент
- **Тяжёлая** (2+ независимых подзадачи по 30+ сек) → параллель

Если не «тяжёлая» — отдай через 1 подагента, не строй оркестратор.

### Шаг 2. Разбей на подзадачи
- 2-7 независимых подзадач
- Каждая ≥30 сек работы
- Изолируемая (подагент может отработать автономно)
- Если подзадачи связаны — последовательно, не параллель
- Каждой подзадаче дай **чёткий role** (архитектор, скептик, исследователь, и т.д.)

### Шаг 3. Выбери тип агента

| Тип | Когда | Инструмент |
|---|---|---|
| Одноразовый саб-агент | 1 задача, не recurring | `sessions_spawn` (mode=run, cleanup=delete) |
| Recurring-агент | Повторяется по расписанию | `cron` + `agentTurn` |
| Именованная сессия | Persistent memory между запусками | `sessionTarget: "session:NAME"` |
| Peer-агент | Друг, рефлексия, обсуждение | `sessionTarget: "session:friend"` |
| Skill-бот | Переиспользуемая процедура | `skill_workshop` create |

### Шаг 4. Шаблоны спавна

**Одноразовый саб-агент**:
```
sessions_spawn({
  task: "Цель. Что выяснить. Формат ответа. Ограничения (не выдумывать, не отправлять в Telegram).",
  taskName: "snake_case_handle",
  label: "Человекочитаемое имя",
  runtime: "subagent",
  model: "minimax/MiniMax-M3",
  thinking: "off",   // off | adaptive для MiniMax-M3
  mode: "run",
  cleanup: "delete",  // или "keep" если нужна persistent сессия
  context: "isolated"
})
```

**Recurring cron**:
```
cron({
  action: "add",
  job: {
    name: "имя-в-kebab-case",
    schedule: { kind: "cron", expr: "...", tz: "Europe/Moscow" },
    sessionTarget: "session:имя",
    payload: { kind: "agentTurn", message: "...", model: "minimax/MiniMax-M3", thinking: "off" },
    delivery: { mode: "announce", channel: "telegram", to: "1038917447" },
    enabled: true
  }
})
```

### Шаг 5. Параллельный запуск
- **Все `sessions_spawn` в одном блоке** — будут запущены параллельно
- НЕ жди результата одного, чтобы запустить следующий
- Push-based auto-announce — результаты придут сами
- **НЕ поллить** (`sessions_list`, `sessions_history`, `exec sleep`) после спавна — задушишь

### Шаг 6. Собери результаты
- Push-event'ы придут как user-message
- Если есть **модератор** (нужен консенсус разных ролей) — спавни его после первого раунда
- Если есть **реакции** (2-й раунд дискуссии) — спавни вторую волну после первой
- Синтез: TL;DR + кросс-выводы + рекомендации

### Шаг 7. Прозрачность
- Сообщи Кариму: «N агентов, Xk токенов, Y мин wall-time, стратегия»
- Если были провалы — скажи

### Шаг 8. Документация
- Создал recurring-агент → добавь в `BOTS.md` таблицу
- Создал skill → проследи, что proposal pending (Карим одобрит)

## Когда НЕ применять
- Лёгкие задачи (adaptive → сам)
- Простой вопрос-ответ
- Уже есть подходящий cron-агент
- Карим сказал «сделай сам» явно

## Стоимость (ориентир)
- 1 подагент: 10-30k токенов
- 2-3 параллельных: 30-80k
- 4+ параллельных: 80-150k
- Recurring cron: 10-30k на запуск

## Связь с другими правилами
- **BOTS.md правило 10 (adaptive)** — главный приоритет, решает делегировать ли вообще
- **BOTS.md правило 7 (асинхронный диспетчер)** — после спавна верни короткий статус
- **BOTS.md правило 9 (прозрачность по токенам)** — в конце отчёта
- **BOTS.md правило 5 (при сомнениях — спросить)** — если не уверен в стратегии

## Граничные случаи
- **Задача требует контекста с Каримом** → делай сам, подагент не увидит историю чата
- **Подзадачи зависят друг от друга** → последовательно, не параллель
- **Подагент упал** → попробуй ещё раз с другим taskName, или сделай сам, или возьми другую стратегию
- **Не уверен, справится ли подагент** → спавни 1, проверь результат, потом добавляй ещё
- **Telegram-канал недоступен** → подагенты не смогут слать в TG, верни результат сам через main-сессию

## Пример: разбор конкурентов (тяжёлая задача)

**Задача Карима**: «Разбери рынок AI-планировщиков».

**Шаги**:
1. Adaptive → тяжёлая (4 подзадачи)
2. Подзадачи: market-scan, competitor-deep, trend-watch, tech-audit
3. Тип: 4 одноразовых subagent
4. Шаблон: см. выше, `taskName: "market-scan"` и т.д.
5. Запуск: 4 `sessions_spawn` в одном блоке
6. Сбор: дождаться 4 push-event'ов, синтезировать
7. Прозрачность: «4 агента × 25k = 100k, 1м 42с wall-time, выигрыш ×2.6»

## Связь с deep-web-research skill

Этот скилл отвечает за **структуру** (кого поднять, как координировать). `deep-web-research` отвечает за **метод** поиска в сети. Используй вместе: при research-задаче поднимай подагентов через `agent-orchestration`, а они применяют `deep-web-research` для сбора данных.
