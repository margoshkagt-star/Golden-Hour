# Task Triage — триаж, декомпозиция и автовыполнение

Применяется, когда пользователь передаёт список задач и хочет получить структурированный, приоритизированный, категоризированный план с декомпозицией сложных пунктов **и автоматическим выполнением того, что агент может сделать сам**.

---

## Когда применять

- Получен список задач (в свободной форме, из трекера, из сообщения, из заметки, из голосового расшифрованного).
- Нужно быстро понять, что делать в первую очередь.
- Нужно разбить крупные задачи на подзадачи.
- Нужно сгруппировать задачи по категориям (включая создание новых категорий, если ни одна существующая не подходит).
- Нужно спланировать день/неделю из бэклога.
- **Нужно, чтобы агент сам выполнил то, что в состоянии выполнить, и свёл к пользователю только требующее его участия.**

---

## Входные данные

Минимально — список задач (даже одна). Желательно, чтобы каждая задача могла содержать:

| Поле | Тип | Комментарий |
|------|-----|-------------|
| `title` | string | Название задачи (обязательно) |
| `description` | string | Подробности (опционально) |
| `deadline` | ISO date | Дедлайн (опционально). Отсутствие дедлайна явно учитывается в формуле веса |
| `importance` | 1–5 / low/med/high/critical | Прямая подсказка от пользователя (опционально) |
| `difficulty` | 1–5 / easy/medium/hard | Уже известная сложность (опционально). Если не указана — оценивается скиллом |
| `context` | string | Контекст (работа/личное/проект X) — опционально |
| `no_decompose` | bool | Пользователь явно просит не разбивать |
| `no_auto_execute` | bool | Пользователь явно просит только спланировать, не выполнять |

Если полей мало — скилл делает выводы по названию, описанию и контексту разговора.

## Алгоритм

### Шаг 1. Категоризация (одна задача → несколько категорий)

> Категоризация идёт **первой**, потому что категория влияет на итоговый вес через `category_boost`.

#### Стандартный пул категорий (расширяемый)

- 🏢 **Работа** — задачи работодателя/заказчика
- 💼 **Проект** — подкатегория создаётся по имени проекта (`проект:app-x`)
- 📚 **Обучение** — курсы, книги, навыки
- 🏥 **Здоровье** — спорт, врачи, режим, сон
- 💰 **Финансы** — платежи, налоги, инвестиции, долги
- 🏠 **Быт** — дом, ремонт, уборка
- 👨‍👩‍👧 **Семья/отношения** — близкие, дети, друзья
- 🧠 **Личное развитие** — привычки, цели, рефлексия
- 🎨 **Хобби/творчество** — рисование, музыка, игры
- 📢 **Коммуникации** — звонки, письма, встречи
- 🛒 **Покупки** — заказы, списки
- ✈️ **Путешествия** — поездки, билеты
- 📋 **Административное** — документы, справки, формальности
- 🗄 **Архив/фон** — задачи без давления, "когда-нибудь"

#### Правила присвоения

1. **Одна задача МОЖЕТ иметь несколько категорий.** Минимум 1, разумный максимум 2–4 (если больше — это сигнал, что задача плохо сформулирована; стоит пометить и при декомпозиции разделить по сути).
2. Мультикатегорийность — **фича, не баг**. Не сворачивать в одну "главную" категорию, даже если кажется логичным.
3. Перед присвоением проверять минимум 3 контекстных маркера, цель — **≥ 2 категории**, если задача не атомарна.
4. Если ни одна стандартная категория не подходит — создать новую с понятным названием и эмодзи.
5. Новые категории сохранять в `memory/task-categories.md` для повторного использования.
6. Перед созданием новой — проверить по смыслу: может, подходит существующая (синонимы, смежные темы).

#### Триггеры на создание новой категории

- Задача явно относится к области, которой нет в списке.
- Пользователь ранее упоминал конкретную сферу ("моя студия", "канал в Telegram", "гараж").
- Задача повторяется 2+ раза в разных формулировках за последние дни.

#### Структура категории

- `name` — короткое имя в нижнем регистре
- `emoji` — 1 эмодзи-маркер
- `keywords` — триггерные слова (пополняются автоматически)
- `boost` — поправка к весу (−2…+2), выставляется пользователем явно или выводится из частоты

---

### Шаг 2. Присвоение веса (важности)

**Вес = итоговая срочность + значимость.** Шкала 1–10 с метками.

#### Формула

```
weight = clamp(1, 10,
    base_importance
  + deadline_score
  + category_boost
  + context_keywords
)
```

#### Компоненты

**`base_importance` (1–5)** — если пользователь явно указал, берётся как есть. Иначе — по описанию:

| Сигнал | Значение |
|---|---|
| "критично", "горит", "срочно", "ASAP", "сегодня" | 5 |
| "важно", "нужно", "в первую очередь" | 4 |
| "хорошо бы", "желательно" | 3 |
| "когда-нибудь", "возможно" | 2 |
| "фоновая", "по возможности" | 1 |

**`deadline_score` (0–5)**:

| Дедлайн | Значение |
|---|---|
| сегодня / просрочен | +5 |
| в течение 1–2 дней | +4 |
| в течение недели | +3 |
| в течение месяца | +2 |
| позже / указан без даты ("квартал", "Q3") | +1 |
| **не указан** | **0** (нейтрально, не минус) |

**`category_boost` (−2…+2)**:

- Категории из списка "критичных для пользователя" → +1…+2 (настраивается явно или выводится из частоты упоминаний)
- Категории, которые пользователь часто откладывает → +1 (фактор забывания)
- Фоновые/низкоценные категории ("почитать когда-нибудь") → −1…−2

**`context_keywords` (−1…+2)**:

- "срочно", "критично", "клиент ждёт", "блокер", "горит" → +1…+2
- "не горит", "когда будет время", "без спешки" → −1

#### Метки веса

| Диапазон | Метка | Эмодзи |
|---|---|---|
| 9–10 | critical | 🔥 |
| 7–8 | high | ⚡ |
| 4–6 | medium | 📌 |
| 2–3 | low | 💤 |
| 1 | archive | 🗄 |

**Если выставлен "наугад"** (мало данных) — пометить `confidence: low`.

### Шаг 3. Оценка сложности и декомпозиция

#### Шкала сложности (1–5)

| Уровень | Описание |
|---|---|
| 1 — тривиально | Одно действие, <15 мин |
| 2 — просто | Несколько шагов, <1 ч |
| 3 — средне | Требует концентрации, до 1 дня |
| 4 — сложно | Много шагов, неочевидные зависимости, до недели |
| 5 — эпик | Требует исследования/планирования, > недели |

#### Когда декомпозировать

- `difficulty ≥ 4` И
- НЕ выставлен флаг `no_decompose` И
- Задача не атомарна по сути ("ответить на письмо", "купить молоко")

#### Принципы декомпозиции

1. **Результат-ориентированно** — каждая подзадача = законченный кусок результата.
2. **Самодостаточно** — подзадача выполняется без знания других (или явно ссылается).
3. **Минимум 3, максимум 7 подзадач** (идеально 3–5). Меньше — не декомпозиция, больше — перегруз.
4. **ПЕРВАЯ подзадача ОБЯЗАТЕЛЬНО должна быть самой маленькой (< 15 мин)** — "разогрев", снижает прокрастинацию. Это **жёсткое правило**, а не рекомендация.
5. **Наследование** — категории и базовый вес наследуются; сложность пересчитывается вниз (после декомпозиции все подзадачи ≤ 3).
6. **Связи** — помечаем зависимости: `blocked_by`, `blocks`.

#### Формат подзадачи

- `title` — действие в инфинитиве ("сформулировать ТЗ", "согласовать с дизайнером")
- `estimated_difficulty` — 1–3
- `acceptance` — **ОБЯЗАТЕЛЬНО**: что считается сделанным (1 строка)
- `blocks` / `blocked_by` — массив id
- `categories` — наследуются
- `weight` — наследуется ± 1
- `capability_hint` — (опционально) подсказка от пользователя: "это для меня / для тебя / для кого-то ещё"

#### Рекурсия

Если подзадача после декомпозиции всё ещё ≥ 4 — декомпозировать рекурсивно, но не глубже 3 уровней (затем пометить `needs_human_split`).

#### Когда НЕ декомпозировать

- Задача атомарна ("ответить", "купить").
- Описание содержит "не разбивать" / "одним куском" / `no_decompose: true`.
- Уже декомпозирована пользователем.

### Шаг 4. Capability assessment + самовыполнение ⭐ НОВОЕ

> Это **ключевое расширение**: после декомпозиции агент оценивает каждую подзадачу на посильность и при возможности **выполняет её сам**.

#### Цель

Свести к пользователю только то, что **реально требует его участия**. Всё, что агент в состоянии сделать сам — сделать и доложить.

#### Capability matrix

🟢 **SELF — выполняется автоматически, без подтверждения:**

- Чтение/запись/редактирование локальных файлов (`read`, `write`, `edit`, `apply_patch`)
- Поиск по памяти и workspace (`memory_search`, `memory_get`, `exec` read-only)
- Web search и fetch (`web_search`, `web_fetch`)
- Генерация текста, кода, анализа, рефакторинг
- Математика, логика, суммаризация
- Создание черновиков (документы, посты, письма) — **только drafts, не отправка**
- Обновление memory-файлов
- Создание skill proposals (статус: pending)
- Чтение shell-команд (`ls`, `cat`, `find`, `grep`)
- Генерация артефактов (image, audio, video) — если провайдер сконфигурирован
- Планирование (`update_plan`, `create_goal`)
- Список сессий, история (read-only)
- Создание/обновление локальных тестовых файлов, фикстур, документации

🟡 **CONFIRM — требует явного "да" от пользователя:**

- Отправка сообщений в каналы (`message` send в Telegram/Discord/и т.д.)
- Публичные посты (соцсети)
- Изменение конфигов (`config.patch`, `apply` для skill/cron)
- Установка пакетов (`npm install`, `pip install`)
- Запуск exec с `elevated: true` или `host=gateway`
- Spawn sub-agents (`sessions_spawn`)
- Изменение `model`/`thinking` через `session_status`
- API-вызовы с side effects
- Создание/изменение cron jobs

🔴 **USER-ONLY — никогда не пытается выполнить, только помечает:**

- Физические действия (сходить, поехать, встретить)
- Финансовые транзакции (оплата, перевод)
- Реальные взаимодействия, требующие личного присутствия
- Авторизация, которой у агента нет (банкинг, чужие аккаунты)
- Деструктивные действия без явного rollback (`rm -rf`, format)
- Любые операции, последствия которых необратимы

#### Алгоритм оценки

Для каждой подзадачи (после шага 3), если `no_auto_execute != true`:

1. **Классификация:** определить 🟢/🟡/🔴 по тексту подзадачи + capability matrix. Ключевые маркеры:
   - 🟢: "написать", "составить", "найти", "проанализировать", "сгенерировать", "создать draft", "обновить память"
   - 🟡: "отправить", "опубликовать", "установить", "изменить конфиг", "запустить агента"
   - 🔴: "сходить", "купить (физически)", "оплатить", "позвонить (голосом)", "встретить", "поехать"
2. **Учёт capability_hint:** если пользователь явно указал — использовать его.
3. **Неуверенность → спросить:** если нельзя однозначно классифицировать — спросить пользователя, не выдумывать.

#### Алгоритм выполнения

```
для каждой подзадачи:
  если status ещё не done:
    если capability == 🟢:
      выполнить()
      если успех:
        status = done
        result = "краткий итог"
        log в task-execution-log.md
      если провал:
        status = failed
        остановить цепочку подзадач этой задачи
        спросить пользователя
    
    если capability == 🟡:
      спросить_подтверждение("Выполнить '{title}'? Это отправит/опубликует/изменит...")
      если да:
        выполнить()
        status = done или failed
        log
      если нет:
        status = skipped
    
    если capability == 🔴:
      status = requires_user
      добавить в очередь пользователя
      (НЕ выполнять)
```

#### Стоп-условия

- **Лимит безопасности:** максимум 5 auto-executions (🟢) за сессию; дальше — стоп, отчёт пользователю.
- **Каскадный провал:** если подзадача провалилась — не выполнять зависимые, пометить как `blocked_by_failure`.
- **Зависимости:** если подзадача `blocked_by` другой невыполненной — ждать.
- **Неуверенность:** если нельзя надёжно классифицировать capability — спросить пользователя, не выдумывать.

#### Логирование

Каждое действие пишется в `users/<user_key>/task-execution-log.md`:

```markdown
# Execution Log

## 2026-06-17 13:35:22
- task: t_001_2 "Сформулировать 3 тезиса"
- capability: 🟢 self
- status: done
- tools: write
- artifact: users/<user_key>/draft-theses.md
- duration: ~30 сек
- result: 3 тезиса сохранены в draft-theses.md
```

#### Безопасность

- 🟢 — выполняется, но лимитировано (5 за сессию)
- 🟡 — **НИКОГДА** не выполняется без подтверждения в этой же реплике
- 🔴 — **НИКОГДА** не пытается выполнить, только маркирует
- Деструктивные команды (`rm`, format, drop) → всегда требуют подтверждения как 🟡
- Любой внешний side effect (отправка, оплата, пост) → лог + подтверждение
- Параметр `no_auto_execute: true` на входе → пропустить весь Шаг 4, только план

#### Что НЕ делать

- Не выдумывать capability: если не уверен → спросить.
- Не выполнять 🔴-задачи, даже если они "простые".
- Не пропускать логирование (каждое действие — в лог).
- Не выполнять длинные цепочки (max 5 за сессию).
- Не вызывать side-effects без подтверждения (даже если "точно знаю, что пользователь хочет").
- Не превышать лимит, даже если осталось "ещё чуть-чуть".

## Память скилла

- `memory/task-categories.md` — общий пул категорий с триггерами и boost (общий для всех пользователей).
- `users/<user_key>/user-priorities.md` — что пользователь считает критичным (по явным подсказкам и истории).
- `users/<user_key>/task-execution-log.md` — лог всех auto-executions (только для аудита, можно чистить раз в месяц).

Шаблон `task-categories.md`:

```markdown
# Категории задач

## 🏢 работа
- keywords: [работа, офис, начальник, заказчик, клиент]
- boost: 0

## 💼 проект:<имя>
- keywords: [...]
- boost: +1
```

Шаблон `user-priorities.md`:

```markdown
# Приоритеты пользователя

## Критичные сферы
- ...

## Часто откладываемые
- ...

## Capability preferences
- max_auto_executions_per_session: 5
- default_auto_execute: true
- excluded_categories_from_auto: []
```

## Первый запуск (cold start)

Если данных мало, задать пользователю 2–3 уточняющих вопроса:

1. "Это всё по работе или есть личное вперемешку?"
2. "Что для тебя сейчас самое важное в жизни/работе? (чтобы настроить category_boost)"
3. "Есть ли что-то, что ты постоянно откладываешь?"
4. **NEW:** "Хочешь, чтобы я сам выполнял то, что могу, или только планировал? (по умолчанию — выполняю безопасное)"

Ответы сохранить в `users/<user_key>/user-priorities.md`.

## Принципы

- **Не врать про уверенность.** Мало данных → `confidence: low`.
- **Не плодить категории.** Новая — только если не лезет ни в одну существующую.
- **Не разбивать тривиальное.** Декомпозиция — нагрузка, применять только когда оправдано.
- **Сохранять авторский смысл.** Если пользователь сам разбил — не передекомпозировать.
- **Мультикатегорийность — фича.** Не сворачивать в одну, даже если кажется логичным.
- **Дедлайн ≠ обязательное поле.** Отсутствие дедлайна — нейтральный сигнал, не минус к весу.
- **Самовыполнение — это помощь, не самоуверенность.** Если не уверен в посильности — спросить.
- **Лимит 5 auto-executions — не наказание, а страховка** от каскадных ошибок.
- **Side-effects = подтверждение.** Любое действие с внешним эффектом (отправка, оплата, пост) — только после явного "да".
- **🔴 — никогда.** Физические/финансовые/необратимые действия — только маркировать.

## Сценарии использования

1. **Входящий список из мессенджера:** "вот мой список на день, разрули" → прогнать целиком, выполнить безопасное, свести остальное.
2. **Одна задача:** "оцени вот эту задачу" → отработать шаги 1–3 для одной.
3. **Планирование дня:** "разбей мне 5 главных на сегодня" → отсортировать по весу, отфильтровать top-N.
4. **Ретриаж:** "пересобери мой бэклог" → прогнать заново с учётом новых данных.
5. **Холодный старт:** нет ни одной категории в памяти → задать 2–3 вопроса, завести `task-categories.md`, потом обработать.
6. **Только план (без выполнения):** пользователь ставит `no_auto_execute: true` или явно говорит "только спланируй" → Шаг 4 пропускается.
7. **⭐ Декомпозиция + auto-execution:** "разбери и сделай что сможешь" — скилл сам выполнит тривиальные и безопасные шаги, требующие внешних действий сведёт к пользователю с явным указанием capability.

## Формат подтверждения для 🟡

Когда нужна подтверждение, использовать **один компактный блок**, не серию вопросов:

```
Перед действием нужно твоё "да":

- 🟡 "Отправить Кариму в Telegram: 'Готово, отчёт отправлен'?"
- 🟡 "Установить пакет express через npm?"

Ок/нет/поправить?
```

Если пользователь отвечает частично ("только первое") — выполнить только подтверждённое, остальные оставить `pending_confirmation`.

## Расширяемость

- **Custom capabilities:** пользователь может в `user-priorities.md` пометить capability_hint для конкретных категорий ("все задачи по проекту X — только планировать, не выполнять").
- **Rate limits:** лимит 5/сессию настраивается через `user-priorities.md` (`max_auto_executions_per_session: N`).
- **Лог-ретенция:** по умолчанию `task-execution-log.md` хранится 30 дней, потом архивируется.
