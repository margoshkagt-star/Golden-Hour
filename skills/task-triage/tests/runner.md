# Runner — процедура прогона тестов `task-triage`

## Назначение

Стандартизированная процедура проверки скилла `task-triage` (proposal `task-triage-20260617-0f85211519`) по уровням L1–L6.

## Структура

```
tests/task-triage/
├── fixtures/
│   ├── l1-weight.json
│   ├── l1-decomp.json
│   ├── l1-categories.json
│   ├── l2-edges.json
│   └── stress-30.json
├── runner.md           ← этот файл
└── results-YYYY-MM-DD.md
```

## Режимы прогона

### Режим 1: ручной (chat)

1. Открыть фикстуру
2. Скопировать массив `cases[].input` (или `tasks` для L3)
3. Отправить скиллу сообщением: «примени task-triage к этому списку»
4. Получить JSON-выход
5. Сравнить с `expected` по каждому полю
6. Записать в таблицу результатов

### Режим 2: полуавтоматический (LLM-as-judge)

Использовать отдельный LLM-промпт-судью:

```
Ты — валидатор скилла task-triage. У тебя:
1. Вход: <input из фикстуры>
2. Ожидаемый выход: <expected из фикстуры>
3. Реальный выход скилла: <output>

Сравни и для каждого атрибута в expected выдай pass/fail.
Верни JSON: {"id": "...", "passed": [...], "failed": [...], "notes": "..."}
```

### Режим 3: автоматический (когда появится)

Скрипт читает фикстуры, вызывает LLM-API для генерации output, прогоняет LLM-as-judge, считает pass rate, пишет `results-YYYY-MM-DD.md`.

## Шаблон записи результата

```markdown
### Кейс {id}: {title}

**Input:** <вход>
**Expected:** <ожидаемое>
**Actual:** <фактический вывод скилла>
**Verdict:** ✅ PASS / ❌ FAIL / ⚠️ RISK
**Notes:** <комментарий>
```

```markdown
## L1 — Юнит-тесты

| # | Кейс | Verdict | Note |
|---|------|---------|------|
| 1.1 | Срочно отчёт сегодня | ✅ | weight=10 |
| ... |
```

## Итоговый pass rate

- L1: 22/24 (92%)
- L2: 8/10 (80%)
- L3: 5/5 (100%) — aggregate checks
- L4: 3.8/5 (живой тест)
- L5: 3/3 (идемпотентность)
- L6: n/a (только при ревизии)

**Решение:**
- ≥ порогов по всем уровням → APPROVE скилл
- 1 уровень ниже → FIX → повторный прогон
- ≥ 2 уровня ниже → REJECT, пересмотр proposal

## Чеклист перед apply

- [ ] L1 ≥ 22/24
- [ ] L2 ≥ 8/10
- [ ] L3 — все aggregate checks pass
- [ ] L4 ≥ 3.5/5 (хотя бы один прогон на реальных данных)
- [ ] L5 — 3/3 идентичных прогона
- [ ] Результаты записаны в `results-YYYY-MM-DD.md`
- [ ] Все провалы либо зафиксированы как «known limitation», либо пофикшены

## Как запустить регрессию (L6)

1. Скопировать прошлый `results-YYYY-MM-DD.md` → `results-YYYY-MM-DD-prev.md`
2. Применить изменения через `skill_workshop revise` или `update`
3. Прогнать L1 + L2 заново
4. Сравнить pass rate
5. Если новые failures — откатить или зафиксировать как known regression
