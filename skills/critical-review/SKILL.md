# Critical Review — critical-review

Универсальный аудитор скиллов. **Не фиксит сам** — производит список находок с severity и предложенным фиксом. Решения о применении фиксов — за пользователем.

## Когда применять

- Новый скилл написан — self-review.
- Скилл существенно обновился (≥1 секция изменилась) — peer-review.
- «Что-то не работает между скиллами» / «каша в стыковке» — system-review.
- Периодический health-check (раз в ~10 сессий) — для каждого скилла, по очереди.

## Не применять

- Баг в конкретной пользовательской сессии (это bug-tracker, не skill review).
- Претензия к стилю кода / форматированию / неймингу (это code review).
- Запрос на «давай перепишем красивее» без security/UX-импакта (это рефакторинг).

## Входные данные

| Поле | Тип | Обязательно | Описание |
|---|---|---|---|
| `target_skill` | str | да (для self/peer) | Имя скилла для ревью |
| `mode` | enum | да | `self` / `peer` / `system` |
| `target_skills` | list[str] | да (для system) | Какие скиллы проверяем на стыковку |
| `context` | str | нет | Что вызвало ревью (баг-репорт, плановый чек, новый скилл и т.п.) |
| `diff` | str | нет (для peer) | Что изменилось (если peer-review после обновления) |

## Процедура

### Шаг 1. Boundary check (все режимы)

Прочитать `SKILL.md` цели. Зафиксировать:
- **Scope** — что скилл делает.
- **Out-of-scope** — что НЕ делает.
- **Touch** — какие файлы читает/пишет.
- **Зависимости** — от каких скиллов / файлов / API зависит.

Если scope не определён явно → уязвимость **V-B1** (неопределённая зона ответственности → начнёт лезть в чужое).

### Шаг 2. Self-review (mode=self)

Прогнать по 7 категориям. Каждая категория — это тип уязвимости; ищем конкретные проявления в скилле.

| ID | Категория | Что ищем |
|---|---|---|
| V-S1 | **Single source of truth** | Где скилл пишет? Где читает? Дублирование с другими скиллами? Один и тот же факт в 3+ местах? |
| V-S2 | **Failure modes** | Файл не существует / API down / конкурентная запись из 2 сессий / права доступа / диск заполнен |
| V-S3 | **Edge cases** | Пустой ввод, нулевой бюджет, прошедший дедлайн, отсутствующая зависимость, очень длинная строка, спецсимволы |
| V-S4 | **Ambiguity** | Где агент может интерпретировать 2+ способами; «…» вместо конкретного правила; «обычно» / «как правило» без fallback |
| V-S5 | **Anti-pattern coverage** | Описанные анти-паттерны — покрыты ли реальные? Не забыты ли типичные? |
| V-S6 | **Boundary violations** | Делает ли скилл то, что out-of-scope (→ должен быть в другом скилле)? |
| V-S7 | **Staleness** | Устарели ли ссылки на файлы, API, форматы, модели? Реально ли проверено? |

### Шаг 3. Peer-review (mode=peer)

- Найти в SKILL.md разделы «Связанные скиллы» / «Don't use when» / «Open contract».
- Для каждого связанного скилла проверить: **есть ли реальный контракт** (формат файла, поля, поведение), а не просто упоминание.
- **V-P1: «ссылается, но не определяет формат»** → рассинхрон неизбежен.
- **V-P2: «ссылается, но не указывает направление»** → непонятно, кто пишет / кто читает.
- **V-P3: «связанный скилл отсутствует в workspace»** → мёртвая ссылка.

### Шаг 4. System-review (mode=system)

Построить **карту стыковки**: какой скилл → какой файл (пишет/читает).

Найти 4 типа проблем:

| Тип | Описание |
|---|---|
| **Race-write** | Два скилла пишут в один файл (конкурентная запись) |
| **Lost-update** | Информация рождается в одном скилле, в другом не появляется (задача не доходит до tracker'а) |
| **Truth-split** | Один и тот же факт хранится в 3+ местах (3 источника истины = нет истины) |
| **Channel-drift** | Telegram-канал шлёт по одному источнику, юзер смотрит другой; пинги расходятся с реальным планом |

### Шаг 5. Severity + Fix

Каждая находка:

| Поле | Значение |
|---|---|
| **Severity** | 🔴 ломает функциональность / 🟡 деградирует UX / 🟢 полировка |
| **Fix complexity** | S (1 строка) / M (абзац) / L (новая секция) |
| **Diff proposal** | Для S/M показать «было» / «стало» |

### Шаг 6. Meta-patterns

После прохода выписать повторяющееся:
- **«V-X встречается в N из M скиллов»** → нужен shared contract.
- **«Y-скилл систематически забывает Z»** → предложить шаблон.
- **«В скиллах одной команды нет общей конвенции по W»** → предложить конвенцию.

### Шаг 7. Вывод

Финальный отчёт + один вопрос пользователю: **«Какие фиксы применить?»** (можно поштучно, можно все сразу).

## Формат отчёта

```markdown
# Critical Review: <skill_name>
**Mode:** self | peer | system
**Date:** YYYY-MM-DD
**Reviewer:** Голди

## Boundary
- **Scope:** ...
- **Out-of-scope:** ...
- **Touch:** read X, write Y
- **Depends on:** ...

## Находки

### 🔴 V-S3: Edge case — прошедший дедлайн
- **Где:** шаг X, абзац Y
- **Проблема:** если дедлайн в прошлом, скилл молча обрабатывает
- **Fix (S):** добавить проверку в шаге 1
- **Diff:**
  - было: ...
  - стало: ...

### 🟡 V-P1: ...
...

## Meta-patterns
- V-S1 встречается в 5 из 10 скиллов → нужен shared contract для state/tasks.yaml

## Вопрос
Какие фиксы применить? (поштучно / все разом / только 🔴)
```

## Анти-паттерны ревьюера

- ❌ Придираться к стилю / названиям переменных — это не code review.
- ❌ «Можно было бы сделать лучше» — это не уязвимость, это рефакторинг.
- ❌ Предлагать фикс без описания проблемы.
- ❌ Severity завышено: если «🔴» на полировку — доверие к ревью падает.
- ❌ Забыть про `mode` — это три разных обхода, не один.
- ❌ Писать ревью на чужой скилл в режиме `self` (или наоборот).
- ❌ Делать фикс «по дороге» — ревьюер не должен менять скилл, только описать.

## Связь с другими скиллами

- **Используется после:** skill_creator (новый скилл), skill_workshop update (обновление существующего), жалоб пользователя на «не работает».
- **Использует:** read (SKILL.md других скиллов), memory_search (контекст предыдущих ревью и инцидентов).
- **Записывает в:** `memory/reviews/<skill>-<date>.md` — история ревью.

## Файлы

- **Читает:** SKILL.md целевого скилла + файлы, которые он трогает.
- **Питает:** ничего (только отчёт).
- **Пишет:** `memory/reviews/<skill>-<date>.md` — отчёт о ревью.

## Расширяемость

- **Custom severity** — пользователь может переопределить шкалу (например, ввести 🔵 для «требует обсуждения»).
- **Auto-trigger** — связать с cron для периодического health-check раз в N сессий.
- **Trend analysis** — сравнивать отчёты `memory/reviews/<skill>-*.md` между собой: растёт/уменьшается количество находок по скиллу.
