## 3. Договорённости по стилю работы с кодом

Сформулированы 16.06.2026 в диалоге, подтверждены Sayu.

### 3.1. Подробные объяснения
Sayu — почти ноль по вебу. Любой технический термин расшифровывается сразу и в контексте. Сравнения с понятными вещами (быт, школа, аналогии из Java-игры, если помогает). Без «само собой разумеется» и без снисходительности.

### 3.2. Поблочный формат, не построчный
Разбираем структуру файла → блок за блоком:
- Что это за блок, зачем нужен
- Внутри блока — расшифровка ключевых классов/атрибутов
- HTML-скелет (doctype, html, head, body) не перечисляется построчно, а подаётся как «служебная рамка»

### 3.3. Альтернативы + обоснование (ОБЯЗАТЕЛЬНО)
Для каждой значимой строчки/блока даю три вещи:
1. **Что делает** — простым языком
2. **Альтернативы** — какие ещё были варианты (другие фреймворки, другие атрибуты, другие подходы)
3. **Почему выбрал именно это** — что выигрываем, что проигрываем, какие trade-offs

Цель: Sayu должна уметь сама принимать осознанные решения в следующий раз, а не копировать «потому что агент так сказал».

### 3.4. Правило «уродливого черновика»
Никакого перфекционизма. На старте НЕ делаем:
- кастомный CSS-файл (хватает Tailwind-классов)
- красивый дизайн (один Tailwind-компонент на страницу — и ладно)
- анимации, тёмная тема, иконки
- GitHub Actions / CI
- собственный домен
- production-grade RLS (для MVP — «все читают, свои пишут»)

Главное: пользователь может зарегаться → войти → задать вопрос → увидеть его в ленте. Это и есть MVP.

### 3.5. Лимит 1.5–2 часа в день
Не сидеть до утра. Признаки усталости: засыпание, раздражение, ошибки в коде. Решения — power nap 20 мин, перерыв, остановка сессии. Лучше недоделать и продолжить завтра, чем сделать «на чистом адреналине» и потом неделю отдыхать.

### 3.6. Без жаргона без расшифровки
Если Sayu видит технический термин без объяснения (CDN, RLS, RLS-policy, implicit flow, build-step, env vars) — это ошибка агента. Объяснять сразу, не откладывая «на потом».

### 3.7. Без воды
Не писать «Это очень важно!» / «Предлагаю начать с…» / «Я уверен, у тебя получится!» / «Отличный вопрос!» / «Как мы знаем…». Только конкретика: что делаем, зачем, как, в каком порядке.

### 3.8. Проверка, что у Sayu уже есть свой код
Прежде чем присылать свой вариант HTML/JS/SQL — спросить, есть ли у Sayu готовый файл (например, от Gemini). Если есть — сначала разобрать его, а не навязывать альтернативу.

## 6. Структура кода, который мы пишем

Целевая структура проекта `qa-site/`:

```
qa-site/
├── index.html         # лендинг + лента вопросов
├── ask.html           # форма нового вопроса
├── question.html      # страница вопроса + ответы
├── login.html         # вход / регистрация
├── profile.html       # профиль + карма
├── js/
│   ├── supabase.js    # createClient(URL, ANON_KEY)
│   ├── auth.js        # signUp / signIn / signOut
│   ├── questions.js   # CRUD вопросов
│   ├── answers.js     # CRUD ответов
│   └── ui.js          # рендер DOM, обработчики
└── README.md          # англоязычный, для MIT-портфолио
```

Все файлы создаются с разбором: что делает + альтернативы + почему именно так.

## 7. Прецеденты (для переиспользования формата)

### Прецедент 1: разбор `<head>` блока
Использован 16.06.2026 при объяснении HTML-черновика.

Формат:
```
1. <meta charset="UTF-8" />
   Что делает: говорит браузеру кодировку файла
   Альтернативы: не писать, windows-1251, ISO-8859-1
   Почему выбрал: универсальный, поддерживает эмодзи + кириллицу
```
Применяется ко всем `<meta>`, `<script>`, `<link>` строкам, Tailwind-классам, и JS-функциям.
