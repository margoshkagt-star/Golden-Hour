# delegate-via-subagents

> Правило-навык: «Когда Sayu просит поресерчить, проверить эссе, улучшить текст, помочь с кодом, разобрать тему — спавни сабагента, а не делай сам в main session.»

## Зачем

Main session — дорогая, контекст общий, время ответа медленное. Сабагент (isolated, фокусированный brief, свой scratch) делает то же самое **дешевле, чище, без загрязнения main-контекста**, и приносит user-facing summary.

## Триггеры (распознавать в сообщении Sayu)

| Фраза-триггер | Тип задачи | Сабагент-роль |
|---|---|---|
| «поресерчи», «найди информацию про», «собери данные по», «разбери тему» | research | `*-researcher` |
| «проверь моё эссе / текст / сочинение», «отполируй», «отредактируй» | essay-review | `*-writer` (review mode) |
| «улучши [текст]», «перепиши», «сделай чище / понятнее» | writing | `*-writer` |
| «помоги написать [X]» | writing | `*-writer` |
| «напиши / сформулируй / собери список / структурируй» | writing/research | `*-writer` или `*-researcher` |
| «помоги с кодом», «напиши скрипт», «почини», «сделай [X] в коде» | coding | `*-coder` |
| «объясни [тему]», «разложи по полочкам» | explainer | `*-explainer` |
| «проверь / проревью [файл]» | review | `*-reviewer` |

## НЕ триггеры (делать самому, НЕ спавнить)

- casual chat, приветствие, эмоциональные темы
- «напомни в X» / «поставь таймер» → **cron**, не сабагент
- «удали/переименуй/покажи/прочитай мой файл» → **прямой tool call**
- «что ты умеешь» / навигация по workspace → **ответ в чат**
- задачи <2 минут (не окупается overhead спавна)
- результат нужен **немедленно в контексте диалога** (сабагент + yield = лишний раунд)
- личные/эмоциональные разговоры

## Алгоритм

1. **Распознать тип** задачи (см. таблицу).
2. **Выбрать шаблон брифа** (см. ниже).
3. **Сформулировать brief** — короткий, конкретный, с deliverable и anti-фантазиями.
4. **Spawn isolated subagent** через `sessions_spawn`:
   - `runtime: subagent`, `mode: run`, `taskName: <role>-<topic>`
   - `model: default` (если не сказано иначе)
   - `cleanup: delete` после завершения (если не нужен persistent session)
   - **omit `context`** — изолированный, не форкать main-транскрипт
5. **Yield** через `sessions_yield` — не крутить poll-цикл.
6. **Получить completion event** — переписать в user-facing summary (формат ниже), **НЕ форвардить raw internal metadata** (по правилу Messaging).
7. **Связать с существующими скиллами** (см. ниже).

## Шаблоны брифов

### Research (`*-researcher`)

```
Ты — researcher для Sayu. Контекст: [краткое].
Задача: [сформулировать].
Deliverable: путь к файлу + формат.
Источники: какие приоритетны (arXiv? GitHub? LinkedIn? news?).
Anti-фантазии: НЕ выдумывать людей/факты, проверять реальность.
Privacy: PII Sayu НЕ писать в output (только обобщённый контекст).
```

### Essay review (`*-writer` mode=review)

```
Ты — редактор эссе. Файл: [путь].
Задача: оценить по [rubric: clarity/voice/structure/specificity/anti-coaching/word-count].
Вернуть: оценки по каждому критерию + 3–5 конкретных улучшений (с цитатами строк).
Не переписывать — только ревью.
```

### Writing (`*-writer`)

```
Ты — writer для Sayu. Цель: [тип текста: essay/abstract/outline/email].
Аудитория: [кто читает].
Тон: [anti-coaching / direct / spec].
Длина: [target words].
Deliverable: файл по пути [путь].
```

### Coding (`*-coder`)

```
Ты — coder для Sayu. Stack: [язык/фреймворк/версии].
Задача: [что сделать].
Тесты: [нужны ли].
Deliverable: путь к коду + (если есть) тесты.
Constraints: [что НЕ делать].
```

### Explainer (`*-explainer`)

```
Ты — explainer для Sayu. Уровень: [IJSO/олимпиадный/университетский].
Тема: [что объяснить].
Формат: [concept-first / example-first / derivation].
Длина: [короткий/средний/полный].
```

## User-facing summary (после yield)

Когда приходит completion event, **переписать** в нормальный voice, **НЕ** форвардить:
- raw internal metadata
- subagent session ids / run ids / token stats
- prompt-data блоки

Формат ответа Sayu:

```
**{Role} завершился. Результат — {одна строка оценки}:**

✅ **Что сделано:**
- {bullet с конкретным deliverable + путём}

⚠️ / 🔴 / 💡 **Что требует внимания:**
- {bullet с риском / open question / red flag}

**Дальше:** {1 строка — что Sayu делать}.
```

## Anti-patterns

- ❌ Спавнить ради мелочей (<2 минут работы)
- ❌ Спавнить если deliverable размыт («подумай над темой»)
- ❌ Спавнить если результат нужен в текущем реплике диалога
- ❌ Спавнить параллельно 2 задачи, пишущие в один файл
- ❌ Форвардить raw subagent output (счётчики токенов, session_id, runtime metadata)
- ❌ Спавнить для cron/reminder (→ `cron` tool)
- ❌ Забыть про privacy: PII Sayu в брифе → обобщать

## Privacy rule (19.06)

В `proposal_content` и в брифе сабагента **НЕ** писать:
- Конкретные названия лицеев / олимпиад
- Имена экзаменов (TOEFL/SAT/ВсОШ/IJSO) — обобщать «олимпиадный уровень», «экзамен по английскому»
- Имена контактов / кружков / клубов

**МОЖНО**: «олимпиадница-старшеклассница», «MIT applicant», «Women in STEM профиль», «стек JS/Python/HTML».

## Связь с другими скиллами

| Скилл | Когда использовать вместе |
|---|---|
| `task-triage` | Перед спавном: разложить задачу на подзадачи, вес, приоритет |
| `note-to-file` | После: сохранить результат в `memory/YYYY-MM-DD.md` + `notes.jsonl` |
| `material-cache` | Для research: сначала проверить `knowledge/`, потом новый search |
| `work-on-site` | Q&A Sayu: код-ревью, разборы, альтернативы |
| `idea-tools` | Если результат — идея, а не факт |
| `show-ideas` | Если Sayu просит «покажи идеи» (агрегация) |
| `critical-review` | Если Sayu просит «проверь скилл / систему» |
| `user-profile` | Перед брифом — прочитать профиль Sayu, чтобы не утекало PII |

## Lessons (обновлять)

- 19.06.2026: создан на основе MIT KG проекта. Сработало: mit-coder, mit-researcher, mit-writer, mit-dedup-monitor — все вернули качественный результат за <10 мин. **Стоимость**: ~6.2M токенов суммарно. **Выигрыш**: main session остался чистым, Sayu получила структурированные deliverables. **Стоит**: продолжать использовать для research/writing/coding/research-review; **не стоит**: для мелких fixes / reminder / casual.
