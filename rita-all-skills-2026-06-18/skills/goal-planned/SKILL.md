# Goal Planned — goal-planned

**Use when:** пользователь приходит с целью (учебной / проектной / олимпиадной / личной) и хочет превратить её в действующий план. Или приносит сразу несколько целей — тогда распределяем между ними общий бюджет времени.

**Don't use when:**
- Цель тривиальная, формальный план не нужен
- План уже есть, нужно только напоминание → `cron`
- Нужно разобрать кашу в голове по текущим мелким задачам → `evening-task-triage`
- Нужен активный список задач на сегодня → `task-tracker` (current-tasks deprecated)

## Назначение

Универсальный планировщик целей с двумя режимами:
- **Режим планирования** — сбор входных, research, 3 подробных черновика, выбор.
- **Режим сопровождения** — ежедневное составление плана на день N (с учётом результатов дня N-1), выдача материалов по запросу, мягкий контроль фокуса.

**Не отключается сам** после выбора плана. Продолжает работать, пока цель активна (до явного «закрываем» / «план завершён» / отмены).

**Не привязан к конкретному пользователю:** никаких hardcoded лимитов или предпочтений. Все параметры запрашиваются.

**Связь с task-tracker:** после выбора плана goal-planned **создаёт стартовый блок задач в `state/tasks.yaml`** (одна запись на milestone). См. шаг 6. Это закрывает дыру «план есть, а в трекере пусто».

## Принципы

### 1. Подробность черновиков

Каждый черновик — это не «идея в одном абзаце», а почти готовый мини-план: фазы с длительностью, разбивка по неделям/дням, объём материала, ресурсы, milestone'ы, формат самопроверки, типичные ловушки, план Б. Три черновика — три разных «мира», каждый из которых пользователь мог бы взять и начать исполнять уже сегодня.

### 2. Удержание на теме (контроль дрейфа)

При работе с целью (планирование, чек-ины, адаптации) **агент активно следит за фокусом** и мягко возвращает к теме, если пользователь уезжает:

- В чек-ине пользователь перечисляет 10 задач, из них 1 — по цели, 9 — побочное → сначала зафиксировать прогресс по цели, потом уточнить, что с остальным.
- В `evening-task-triage` появляются задачи, конкурирующие с бюджетом цели → пометить конфликт, не дать сьехать в «аа, потом разберусь».
- На адаптации пользователь говорит «давай переключимся на X» → спросить «это новая цель или относится к текущей?», при необходимости запустить `goal-planned` заново.
- Жёсткого запрета нет — это не тюрьма. Но по умолчанию фокус на текущей цели до явного «закрываем».

**Как сигналить пользователю:** короткая реплика вроде «Замечу — это в сторону от цели по физике. Сначала закроем milestone X, потом разберёмся?». Без нравоучений, без морализаторства.

### 3. Агент не отключается сам

После выбора плана и записи в файл навык **не уходит в «фон»**. На каждой итерации (новый день / запрос «что делать сегодня?» / отчёт о вчерашнем) он:

- Подхватывает контекст цели (читает файл плана).
- Учитывает факт/план по предыдущим дням.
- Выдаёт план на текущий день N в формате дня 1.
- Готов к адаптации, если вчера не успели.

«Тишина» между днями ≠ отключение. Следующий контакт по цели = следующая итерация.

### 4. Материалы — по запросу, не в плане

План дня — это **что делать** (задачи по часам, конкретные действия, ожидаемый результат), а **не учебник**. Никаких видео, ссылок, разборов, текстов задач, шпаргалок внутри таблицы плана. Иначе он распухает и перестаёт быть рабочим инструментом.

**Правило для агента:**
- При **первой выдаче плана** (день 1) и при **начале новой сессии по цели** — явно сказать пользователю: «материалы выдам по запросу — скажи "дай материал для задачи X" / "покажи разбор" / "ссылку"».
- Дать **мини-меню типовых запросов** (чтобы юзер в стрессе не мучился формулировкой): «разбери задачу X» / «дай теорию по теме Y» / «покажи пример решения» / «найди видео по Z».
- **Не выдавать** материалы проактивно. Только по явной просьбе.
- Если пользователь сам просит «включи всё в план» — окей, включить, но это исключение, а не правило.

**Что может быть в плане (без материалов):**
- «Решить 3 задачи на закон Кулона» ✓
- «Прочитать §2 Мякишева, выписать формулы» ✓
- «Посмотреть 1 видео по теме, выбрать лучшее» — упоминание видео как действия, но без ссылки

**Что НЕ должно быть в плане (материалы):**
- ❌ Сама ссылка на видео
- ❌ Текст задачи
- ❌ Полный разбор
- ❌ Готовые формулы в шпаргалке (это артефакт работы, а не план)

**Граница «источник vs контент»:**
- В плане ОК: **указатели на источник** — «§2 Мякишева», «глава 4», «раздел про ёмкость», «любое видео по теме X на YouTube». Это ссылки на внешнее, не сам контент.
- В плане НЕ ОК: **контент** — конкретная ссылка, текст задачи, готовый разбор, заполненная шпаргалка.
- В файле плана в секции `Research-сводка` — ОК любые ссылки и цитаты (это архив, не рабочий план).

## Входные данные

### Глобальный параметр (запрашивается один раз в начале)

- **Бюджет времени** — сколько часов в день (или в неделю) пользователь готов тратить на ВСЕ цели суммарно. Не задаётся заранее, спрашиваем.
- **Единицы бюджета:** если юзер назвал ч/неделю — нормализовать в ч/день одним из способов:
  - **Поровну:** ч/неделю ÷ 7.
  - **Рабочие дни:** ч/неделю ÷ 5 (Пн–Пт плотно, выходные 0).
  - **Гибкое:** «5 дней × N ч, 2 дня × 0» или похоже.
  - Уточнить у юзера, какой режим, и зафиксировать.

### На каждую цель (4 поля + язык)

| Поле | Что значит | Формат |
|---|---|---|
| **Вес** | Приоритет; при нескольких целях время распределяется пропорционально | Число 1–10 или low/med/high. **Валидация:** должен быть > 0. Если 0 или отрицательный — спросить, не выдумывать. |
| **Дедлайн** | Контрольная дата | Дата. **Валидация:** не в прошлом. Если в прошлом — спросить: сдвинуть или закрыть цель. |
| **Сложность** | Текущий уровень пользователя по теме | Свободный ввод: что знаю/не знаю, опыт |
| **Категория** | Тип цели | Строка: физика, программирование, язык, олимпиада… |
| **Язык** | Язык материалов и формулировок | ru / en / mixed. Использовать для research и для текста в плане дня. |

### Бюджет = 0 (edge case)

Если юзер говорит «у меня 0 ч/день на эту цель» — не строить план. Ответить одной строкой: «с 0 ч/день план сделать нельзя. Хочешь минимальный 30 мин/день, или это ошибка ввода, или пока зафиксировать цель как `на паузе`?»

### По материалам (уточняется по каждой цели)

- **Свои** — учебники, конспекты, лекции, что уже есть
- **Внешние** — подобрать через research, если попросят

## Процедура

### 1. Сбор входных

- Спросить глобальный бюджет времени (если не дан). Если в ч/неделю — уточнить режим распределения (поровну / рабочие дни / гибкое).
- На каждую цель — 4 поля + язык. Если не даны — спросить по одному, максимум 1–2 уточнения за раз.
- Materials: спросить, что у пользователя уже есть по каждой цели.
- Валидация (см. «Входные данные»): вес > 0, дедлайн не в прошлом, бюджет > 0.

### 2. Research (по каждой цели)

**Шаг 2.0: Проверка готового research-файла.**

Прежде чем делать новый research через `web_search` — **проверить, нет ли уже готового файла**:

| Где искать | Когда использовать |
|---|---|
| `data/research/<topic-slug>.md` | Стандартное место для research-файлов |
| `knowledge/<subject>/<slug>.md` | Если пользователь ссылается на материал из `material-cache` |
| `~/.openclaw/workspace/plans/<slug>.md` (если уже есть) | Перечитать research-сводку из существующего плана |
| Файл, который пользователь явно назвал | «возьми research из `~/path/to/file.md`» |

**Если research-файл найден:**
1. Прочитать целиком.
2. **Парсить ссылки** (см. парсер ниже) — собрать в `research_links: [{url, title, source_type, date, reliability}]`.
3. **Парсить задачи** (если есть) — секции `## Задачи` / `## Tasks` / `- [ ] ...` / `## План действий` и т.п. — собрать в `research_tasks: [{title, source, priority?}]`.
4. Сообщить пользователю: «Нашёл готовый research в `<path>`. Использовать его (Y) или сделать новый (N)?».
5. **При подтверждении** — перейти к Шагу 2.5 (валидация готового research) вместо полного research через web.

**Парсер markdown для ссылок** (финальная версия, v2, исправлены 5 багов):

```javascript
function parseMarkdownLinks(text) {
  const re = /\[([^\]]+)\]\(([^\)]+)\)/g;
  const out = [];
  let m;
  while ((m = re.exec(text)) !== null) {
    let url = m[2].trim().replace(/["',.]+$/, '');  // strip trailing punctuation
    out.push([m[1], url]);
  }
  return out;
}

function parseBareUrls(text) {
  // убрать YAML frontmatter (URL в sources не должны ловиться)
  let t = text.replace(/^---\n[\s\S]*?\n---\n/, '');
  // 1) убрать code blocks
  t = t.replace(/`[^`]*`/g, '');
  // 2) убрать markdown links (чтобы не дублировать)
  t = t.replace(/\[([^\]]+)\]\(([^\)]+)\)/g, '');
  // 3) убрать содержимое внутри скобок (...) (URL в скобках = markdown)
  t = t.replace(/\([^)]*\)/g, '');
  // 4) убрать содержимое внутри [..]
  t = t.replace(/\[[^\]]*\]/g, '');
  // 5) bare URLs
  const urls = t.match(/https?:\/\/[^\s\)\]<>`]+/g) || [];
  // 6) strip trailing punctuation + dedup
  return [...new Set(urls.map(u => u.replace(/["',.]+$/, '')))];
}
```

**Парсер задач:**

```javascript
function parseTasksFromSection(text) {
  const re = /-\s*\[([ xX])\]\s*(.+)/g;
  const out = [];
  let m;
  while ((m = re.exec(text)) !== null) {
    const done = m[1].toLowerCase() === 'x';
    let title = m[2].trim();
    // strip markdown link wrapper
    const linkMatch = title.match(/^\[([^\]]+)\]\(([^\)]+)\)$/);
    if (linkMatch) title = linkMatch[1];
    // extract priority
    let priority = null;
    const prioMatch = title.match(/\(priority:\s*(high|medium|low|h|m|l)\)/i);
    if (prioMatch) {
      const p = prioMatch[1].toLowerCase();
      priority = { h: 'high', m: 'medium', l: 'low' }[p[0]] || p;
      title = title.replace(prioMatch[0], '').trim();
    }
    out.push({ done, title, priority });
  }
  return out;
}
```

**Парсер секций** (нормализует имена, убирает суффиксы в скобках):

```javascript
function parseTaskSections(text) {
  const sections = {};
  let current = null;
  for (const line of text.split('\n')) {
    const m = line.match(/^##\s+(.+)/);
    if (m) {
      current = m[1].trim().toLowerCase().split('(')[0].trim();
      sections[current] = sections[current] || [];
    } else if (current && line.trim()) {
      sections[current].push(line);
    }
  }
  return sections;
}
```

**Валидация age** (использует `frontmatter.updated` или `frontmatter.created`, не mtime файла):

```javascript
function validateResearchAge(filePath, maxAgeMonths = 6) {
  const fm = parseFrontmatter(fs.readFileSync(filePath, 'utf-8')) || {};
  const dateStr = fm.updated || fm.created;
  let date = fs.statSync(filePath).mtime;
  let source = 'mtime';
  if (dateStr) {
    date = new Date(dateStr);
    source = 'frontmatter';
  }
  const ageDays = Math.floor((new Date() - date) / 86400000);
  const isOld = ageDays > 30 * maxAgeMonths;
  return { source, ageDays, isOld, warning: isOld ? `[возможно устарело, ${ageDays} дней, source: ${source}]` : null };
}
```

**Шаг 2.5: Валидация готового research.**
- Дата файла < 6 месяцев? Если старше — пометить `[возможно устарело, требует проверки]`.
- Есть секция с источниками/ссылками? Если нет — предупредить: «файл без явных источников, использую как есть, но качество низкое».
- Соответствует ли теме цели? Спросить пользователя: «research по теме X, у тебя цель по Y — подходит?».

**Шаг 2.10: Новый research (если готового нет или пользователь хочет новый).**

**Главное правило: main НЕ делает research сам. Всегда делегировать субагенту `researcher`.**

Зачем:
- **Изоляция контекста** — research не засоряет main-сессию.
- **Параллелизм** — main продолжает диалог с пользователем, research идёт в фоне.
- **Экономия** — researcher на той же модели `minimax/MiniMax-M3` (OpenRouter-ключ недоступен, вариантов нет), main — на ней же. Изоляция всё равно полезна: не засоряем main-контекст, можем гонять несколько research параллельно.
- **Чистый fallback** — ошибка researcher'а не ломает main.

**Шаг A. Сформулировать бриф для субагента:**
- Название цели, категория, дедлайн, вес.
- Бюджет времени (ч/неделю).
- Язык материалов (ru / en / mixed).
- Список материалов, которые у пользователя уже есть (из входа «Materials»).
- 2–5 конкретных research-вопросов (не «всё про X», а «curriculum / roadmap / формат экзамена / типичные задачи / ресурсы»).
- Требования к факт-чеку (см. ниже).
- Никакого личного контекста пользователя (researcher изолирован — бриф, не болтовня).

**Шаг B. Спавн субагента:**

Требует `openclaw gateway restart` после правки конфига. Если не работает — main спрашивает пользователя и предлагает создать нового субагента вручную.

**Шаг C. Задача субагента (что он делает):**
1. `web_search` + `web_fetch` по research-вопросам (открытые источники: curricula, roadmaps, syllabus, разборы задач, типичные ошибки).
2. **Факт-чек** (обязательно, прописан в брифе):
   - Дата публикации источника < 2 лет. Если старше — пометить `[требует проверки]`.
   - Подтверждение из 2+ независимых источников. Если один — `[единичный источник]`.
   - Для олимпиад/экзаменов/квалификаций — обязательно официальный сайт (ФИПИ, вуз, оргкомитет).
   - Итог: ссылка с датой и оценкой надёжности, либо пометка «не подтверждено».
3. **Сохранить результат в `~/.openclaw/workspace/knowledge/<subject>/<goal-slug>.md`** с frontmatter:
   ```yaml
   ---
   type: research
   subject: <subject>             # physics, math, english, ...
   topic: <goal topic>
   goal_slug: <goal-slug>
   created: <YYYY-MM-DD>
   updated: <YYYY-MM-DD>
   version: 1
   source: web_search+web_fetch
   sources:
     - url: "https://..."
       title: "..."
       accessed: <YYYY-MM-DD>
       reliability: high|medium|low
   quality: <high|medium|low>     # итоговая оценка research
   language: <ru|en|mixed>
   tags: [...]
   ---
   ```
   Структура тела файла:
   - **TL;DR** (1–2 строки)
   - **Суть** (5–10 строк)
   - **Детальный разбор** по research-вопросам
   - **Ссылки** с оценкой надёжности
   - **Темы/задачи** из research
   - **Что не покрыто** (если есть пробелы — чтобы не искать повторно)
4. **Вернуть в финальный ответ:**
   - TL;DR (1–2 предложения)
   - 3–5 главных тезисов
   - Путь к сохранённому файлу
   - Оценку качества (`high` / `medium` / `low`) + причину

**Шаг D. Что main делает после получения результата:**
1. **Проверить, что файл существует** (`read` или `exec ls`). Если нет — fallback (см. ниже).
2. **Сохранить копию в Obsidian** в `Для openclaw/research/<goal-slug>.md` (полный путь — из `TOOLS.md`). Это **отдельная архивная заметка**, привязанная к цели, доступная для повторного использования.
   - Шаблон заметки: frontmatter (аналогично `knowledge/`-файлу) + TL;DR + тезисы + ссылки + ссылка-дубль на оригинал в `knowledge/`.
   - Территория «Для openclaw» — пишем без подтверждения.
3. Обновить `knowledge/<subject>/_index.md` — добавить новый slug.
4. **Записать в файл плана** секцию `## Research-сводка` (шаблон ниже).
5. Сообщить пользователю: «Research готов, сохранён в `knowledge/` и Obsidian. Качество: `<high/medium/low>`. Причина: `<...>`».

**Шаблон Research-сводка в файле плана** (после делегирования):

```markdown
## Research-сводка
_Источник: knowledge/<subject>/<goal-slug>.md (обновлён: YYYY-MM-DD, version: 1, quality: high|medium|low)_
_Архив: Obsidian → Для openclaw/research/<goal-slug>.md_

```

**Fallback (если субагент провалился):**
- Файл не создан / `quality: low` / timeout / `researcher` не сконфигурирован — спросить пользователя:
  - **(a)** повторить (ре-спавн с тем же или уточнённым брифом)
  - **(b)** пользователь даст research руками (свои ссылки / файлы) — main парсит и сохраняет в `knowledge/`
  - **(c)** продолжить без research — пометить в файле плана `research: не выполнен (<причина>)`
- **Main НЕ делает research сам** — это нарушение главного правила шага 2.10.

**Anti-паттерны (для шага 2.10):**
- ❌ Main делает `web_search` / `web_fetch` напрямую (обход правила)
- ❌ `context: "fork"` для researcher'а (fork даст ему main transcript)
- ❌ Poll результата через `sessions_list` / `sessions_history` (push-based, не poll)
- ❌ Несколько researcher'ов на одну цель (один goal = один research)
- ❌ Использовать `openrouter/auto` или другой провайдер вместо `minimax` для researcher'а (ключ OpenRouter не работает)
- ❌ Забыть сохранить копию в Obsidian (`Для openclaw/research/...`)
- ❌ Сказать «план сохранён», не проверив, что `knowledge/<subject>/<slug>.md` создан субагентом
- ❌ Использовать тот же `taskName` для разных целей (потеря истории)
- ❌ Передавать в бриф личный контекст пользователя (researcher изолирован)

### 3. Подробные черновики (≥2, лучше 3)

Обязательно с альтернативами: разные подходы, темп, scope. Каждый черновик содержит следующие блоки:

- **Суть подхода (1–2 предложения):** как именно идём к цели, в чём ключевая идея.
- **Фазы:** таблица — номер, название, что внутри, длительность.
- **Разбивка по неделям/дням:** что делается, какие занятия, в каком порядке.
- **Конкретные темы / модули:** список того, что покрывается (темы, главы, функции, навыки).
- **Объём материала:** цифры — страницы, главы, часы видео, кол-во задач, километры, повторения.
- **Ресурсы по фазам:** какие книги, сайты, видео, инструменты открываем на каждой фазе.
- **Milestone'ы (измеримые):** чекбоксы с датами.
- **Критерии готовности фазы:** как пользователь сам поймёт, что фаза закрыта.
- **Формат самопроверки:** как проверяем прогресс.
- **Типичные ловушки и противодействие:** что обычно мешает + как обходить.
- **Дневная нагрузка:** ... ч. **Примечание:** если нагрузка меняется по фазам — указать диапазон или таблицу по фазам.
- **Плюсы / минусы:** trade-off'ы.
- **План Б при отставании:** что делать, если отстали на N дней (конкретные шаги).

**Самопроверка черновика перед выдачей:** все 14 блоков заполнены? Если «…» / «TBD» — дозаполнить, не выдавать заглушку.

### 4. Расчёт дневной нагрузки

- Бюджет делится между целями пропорционально весам.
- На каждую цель: `(бюджет_в_день × вес_цели / сумма_весов)`.
- Если цель одна — весь бюджет ей.
- **Валидация:** если сумма весов = 0 — спросить юзера (см. «Входные данные»).
- Показать формулу и логику, не магическое число.

### 4.5. Realism check (согласование цели с бюджетом и сроком)

**Добавлен 2026-06-19 после тестов 5 субагентов.** Capacity check (шаг 5) проверяет, что план **влазит в бюджет** — но не проверяет, что бюджет **достаточен для самой цели**. Realism check закрывает эту дыру.

**Эвристики для типовых целей** (хранить в `knowledge/_realism/<category>.md`, обновлять через research):

| Категория цели | Типичный объём | Реалистичная оценка |
|---|---|---|
| `language + CEFR level` | 100-200 ч на уровень (FSI/CLTR) | Испанский A0→B1: 350-400 ч; A0→A2: 150-200 ч; A0→B2: 600-700 ч |
| `ege + 85+ баллов` | 350-500 ч систематической подготовки | Зависит от baseline; <200 ч = нереалистично для 85+ |
| `vsosh + призёр мун.` | 200-400 ч от базы (продвинутый) до этапа | Продвинутый + 4 мес × 15 ч = 240 ч = пограничный кейс |
| `mvp + web` | 80-200 ч для простого MVP | 8 нед × 8 ч = 64 ч — мало для полноценного MVP с auth+AI |
| `fitness + 5 км` | 6-12 нед по 3-4 тренировки/нед | Стандартный план |

**Алгоритм:**
1. Определить категорию цели (`category` из входных).
2. Прочитать `knowledge/_realism/<category>.md` (если есть). Иначе — общие эвристики.
3. Сравнить `total_budget = hours_per_week × weeks_to_deadline` с типичным объёмом.
4. **Вердикт:**
   - `realistic`: бюджет ≥ 80% типичного → OK
   - `tight`: бюджет 50-80% → пометить, предложить увеличить или сократить scope
   - `ambitious`: бюджет < 50% → честно сказать «при таком бюджете цель вряд ли достижима в срок», предложить 2-3 варианта:
     - (a) увеличить бюджет (больше ч/нед)
     - (b) сдвинуть дедлайн
     - (c) сократить scope (напр. не B1, а A2)
     - (d) принять как «амбициозную цель» и работать по максимуму
5. **Записать в файл плана** секцию `## Realism check`:
   ```markdown
   ## Realism check
   - **Категория:** <категория>
   - **Типичный объём для цели:** <N ч>
   - **Твой бюджет:** <total_budget ч> = <hours_per_week> ч/нед × <weeks> нед
   - **Вердикт:** <realistic / tight / ambitious>
   - **Если tight / ambitious:** <рекомендации>
   - **Источник:** knowledge/_realism/<category>.md (если есть) | общая эвристика
   ```
6. **Сообщить пользователю:**
   - `realistic`: не упоминать
   - `tight`: «Цель влезает в бюджет впритык. Рекомендую иметь 20% буфер.»
   - `ambitious`: явно предупредить, дать выбор из 4 вариантов

**Структура `knowledge/_realism/<category>.md`:**
```yaml
---
type: realism-data
category: <category>
updated: YYYY-MM-DD
sources:
  - url: "..."
    accessed: YYYY-MM-DD
data:
  - level: <baseline>→<target>
    hours_min: <N>
    hours_max: <M>
    hours_typical: <avg>
    source: FSI|CLTR|official|community
    note: <краткое пояснение>
```

### 5. Capacity check

- Если требуемая нагрузка > бюджета — флажок «не влезет при заданном бюджете».
- Варианты реакции: продлить таймлайн / увеличить бюджет / урезать scope.
- **При сильном недовлезании** (план требует >1.5× бюджета) — обязательно пометить, что план рискованный, и предложить урезание.

**Edge case: дедлайн слишком близок (triage-режим).**
- **Если до дедлайна <24 ч** — это не план, а triage. Вместо полной процедуры (research → 3 черновика → выбор) агент:
  1. Спрашивает: «Дедлайн через <X ч. Полный план сделать не успеем. Хочешь triage — 3 самых важных действия на оставшееся время, без черновиков и фаз?»
  2. При согласии — выдаёт 3 действия с конкретным временем и сразу план на оставшиеся часы.
  3. При несогласии — план строится как обычно, но с пометкой «цель вряд ли достижима в срок».
- **Если дедлайн в прошлом** — сказать честно: «Дедлайн уже прошёл. Хочешь сдвинуть на новую дату или закрыть цель?»

### 6. Запись плана + Handoff в task-tracker

**Шаг A. Записать план в файл.**
- Путь: `~/.openclaw/workspace/plans/<категория>-<слаг>.md` — один файл на цель.
- **Перед записью:** убедиться, что папка `plans/` существует (`mkdir -p` если нет). На Windows путь = `%USERPROFILE%\.openclaw\workspace\plans\`.
- **Слаг:** транслит категории + 3-5 слов из названия цели, латиницей или кириллицей, без спецсимволов и пробелов (заменять на `-`). Пример: `plans/ucheba-fizika-elektrodinamika-nedelya.md`. **Если файл с таким слагом уже есть** — спросить: «это та же цель (продолжаем) или новая (тогда другое имя)?»
- Содержимое: входные данные, research-сводку, черновые планы, выбранный, дневную нагрузку, историю адаптаций, журнал планов по дням, drift log, handoff-секцию.
- **После записи:** проверить, что файл существует. Если ошибка FS (прав нет, диск заполнен, путь недопустим) — сообщить юзеру и попросить помощь, **не утверждать, что план сохранён**, пока не проверено.
- Несколько целей = несколько файлов.
- Не плодить дубли при обновлении: править существующий файл, а не создавать новый.
- **Структура файла** — секции: Метаданные, Входные, Materials, Research-сводка, 3 Черновика (полные), Выбранный план, Текущая фаза и milestone, Журнал планов по дням, Drift log, Capacity check, История адаптаций, Handoff в task-tracker.

**Шаг B. Сгенерировать стартовый блок задач в `state/tasks.yaml`.** ⭐ Handoff в task-tracker.

После записи файла плана — **обязательно** создать записи в `state/tasks.yaml` для отслеживания. Без этого task-tracker и goal-checkin-notifier не узнают о цели.

Алгоритм:
1. Прочитать `~/.openclaw/workspace/state/tasks.yaml` (создать с пустым `tasks: []` если не существует).
2. Для **каждого milestone из выбранного черновика** создать запись:
   ```yaml
   - id: <auto-increment от max id + 1>
     name: "<milestone name>"
     category: <категория цели>
     weight: <вес цели × 1.0 для M1, ×0.9 для M2, ... — убывание>
     deadline: <дата milestone из черновика>
     status: planned
     progress: 0
     actual_duration: null
     goal_slug: <слаг цели>     # ← связь с plans/<slug>.md
     milestone_id: M1
     source: "goal-planned"
   ```
3. Записать обновлённый `tasks.yaml` (atomic write через temp file + rename).
4. Сообщить юзеру: «План записан + N задач добавлено в трекер. Дашборд обновится в Obsidian автоматически.»

**Если файл `state/tasks.yaml` не существует** — создать с шапкой:
```yaml
tasks: []
meta:
  work_start: "09:00"
  work_end: "21:00"
  updated_at: <ISO datetime>
```

**При адаптации (шаг 7) или изменении цели** — обновить соответствующие записи в `state/tasks.yaml` (новые milestone'ы добавить, закрытые отметить `status=done, progress=100`).

### 7. Адаптация при срыве дедлайна

- **Триггер:** пользователь говорит «сорвал дедлайн X» / «не успеваю» / «план не работает», ИЛИ отставание по плану перешло в категорию «системное» (>3 дней, см. шаг 10).
- **Действия:**
  1. Прочитать файл плана + журнал дней + соответствующие записи в `state/tasks.yaml`.
  2. Сравнить факт/план (что сделано, что нет).
  3. Предложить 2–3 варианта коррекции:
     - Продлить таймлайн
     - Урезать scope (какие milestone'ы можно отложить)
     - Увеличить бюджет
     - Переставить порядок фаз
  4. Если план нереалистичен в текущих рамках — сказать честно, не обещать чуда.
  5. **После любой адаптации (даже перестановки фаз) — явно обновить:**
     - В файле плана: активный milestone (если сдвинулся), дневную нагрузку, даты milestone'ов.
     - В `state/tasks.yaml`: новые дедлайны / статусы для затронутых записей (через `goal_slug`).
  6. Записать новую версию в файл плана с датой адаптации.

**Связь с шагом 9:** после шага 7 следующая выдача плана дня N идёт уже по обновлённой фазе/milestone'ам. Агент обязан перечитать файл и не полагаться на то, что помнит «старое» состояние.

### 8. Контроль дрейфа (на каждой сессии по цели)

- В начале сессии / при чек-ине: «Сначала по цели — что сдвинулось, потом остальное».
- Если пользователь приносит задачи, явно не относящиеся к текущей цели — короткое уточнение: «Это новая цель, дополнительная задача в текущей, или побочное?».
- Если новая цель — **немедленно пересчитать дневную нагрузку текущей цели** (по формуле шага 4 с учётом нового веса). Возможно, потребуется адаптация (шаг 7) текущей цели. Затем запустить `goal-planned` для новой цели.
- Если цели конфликтуют (например, «интенсивный французский за 2 нед» + «начать испанский») — пометить конфликт явно, спросить, какая приоритетнее.
- Если побочное — записать в `evening-task-triage` или отложить, не размывая бюджет текущей цели.
- **Drift log:** каждое drift-событие фиксировать в файле плана (секция `Drift log`): дата, что случилось, как среагировали. Это не «история адаптаций» — это отдельный лог дрейфа.
- Без нравоучений. Просто структура: «вот текущий фокус, вот куда ты смотришь, давай выровняем».

### 9. Ежедневный план на день N (режим сопровождения)

**Триггеры (любой из):**
- Начало нового дня по цели.
- Запрос пользователя: «что делать сегодня?» / «план на сегодня» / «давай день N».
- Отчёт пользователя о вчерашнем дне («вчера сделал только X»).
- Импульс агента: если юзер вернулся к цели впервые за >6 ч и есть открытые дни в журнале — предложить план на текущий день.

**Действия:**
1. **Определить N:** посчитать записи в `Журнал планов по дням` в файле плана. `N = кол-во_записей + 1`. Если журнал пуст → `N = 1`. Эта запись ещё не существует на момент вычисления.
2. **Определить текущую фазу:** последняя фаза, у которой (а) milestone'ы ещё не все закрыты ИЛИ (б) дедлайн фазы ещё не прошёл. Берём то, что «стоп-сигналит» позже. Если впереди — двигаемся; если позади — застреваем в текущей, не прыгаем в следующую.
3. Прочитать файл плана + журнал предыдущих дней + соответствующие записи в `state/tasks.yaml` (по `goal_slug`).
4. Определить:
   - Активный milestone (первый незакрытый в текущей фазе).
   - Факт/план по предыдущим дням (что успели, что нет).
5. **Если факт/план по предыдущим дням показывает отставание — скорректировать день N под реальность (см. шаг 10).**
6. Составить **план на день N** в формате дня 1: таблица «время — что делать — конкретика» + цель дня + итог + куда идём завтра.
7. **Проверить сумму часов** в таблице: она должна быть ≤ дневной нагрузке из выбранного черновика. Если больше — сократить задачи или пометить каждую лишнюю как `[вне бюджета: +X ч]`.
8. **Не прикладывать материалы** в план (см. принцип 4). Материалы выдаются только по запросу.
9. **Обязательно сообщить правило** о материалах (при первой выдаче или начале новой сессии): «материалы выдам по запросу — скажи, если что-то нужно» + мини-меню типовых запросов.
10. **Если активных целей >1** — план на день выдаётся **единым сводным**, с разбивкой по целям (например, утро — цель 1, вечер — цель 2). Сумма часов по всем целям в один день ≤ общему бюджету.
11. Записать план в файл (секция «Журнал планов по дням → День N»). Если multi-goal — отметить цели в каждой строке таблицы (отдельная колонка «Цель»).
12. **Если юзер не отчитался за вчера** (в журнале за день N-1 «Итог дня (факт)» пуст) — одной строкой спросить: «Что вчера закрыли из плана? Или сказать `пропустить`, чтобы двигаться дальше?»
13. **Обновить `state/tasks.yaml`:** для активной milestone-записи проставить `status=in_progress, progress=текущий/100`. Это позволяет дашборду в Obsidian показывать актуальный прогресс.

### 10. Адаптация дневного плана

- Если пользователь сообщает «вчера сделал только X из Y» или «сегодня не успеваю N часов»:
  1. Прочитать журнал дней.
  2. Классифицировать отставание:
     - **Лёгкое (1 день):** перенести часть задач на завтра, общий дедлайн не двигать.
     - **Среднее (2–3 дня подряд):** урезать scope — отбросить низкоприоритетные задачи из активной фазы, общий дедлайн сдвинуть на 1 неделю.
     - **Системное (>3 дней подряд):** пересмотреть план целиком, продлить общий дедлайн, при необходимости запустить адаптацию (шаг 7).
  3. Зафиксировать в истории адаптаций.
  4. Сразу выдать скорректированный план на текущий день (если применимо) или на следующий.
  5. **Обновить `state/tasks.yaml`:** `actual_duration` для завершённых, `progress` для частично сделанных, `status=blocked` для перенесённых.

**Связь с шагом 7:** «Системное» отставание из шага 10 автоматически запускает шаг 7 (адаптация при срыве дедлайна).

### 11. Долгое отсутствие юзера

- Если юзер не возвращался к цели **>7 дней**:
  - Не авто-паузить (решает юзер).
  - При следующем контакте одной строкой: «Прошло N дней, план в режиме ожидания. Возобновляем как есть / сдвигаем дедлайн / закрываем?»

## Правила переходов статусов

Файл плана имеет поле `Статус: активен / на паузе / завершён / отменён`. Правила переходов:

| Из → В | Триггер | Что делать с файлом |
|---|---|---|
| активен → завершён | Все milestone'ы закрыты И юзер подтвердил «готово» ИЛИ (дедлайн прошёл + 100% milestone'ов) | Файл остаётся в `plans/`, статус обновляется, история финализируется. **В `state/tasks.yaml` все записи этой цели → `status=done, progress=100`.** |
| активен → завершён (с провалом) | Юзер сказал «сделал, что мог» при частичных milestone'ах | Файл остаётся, статус «завершён», в истории — пометка «частично: X из Y milestone'ов». **В `state/tasks.yaml` незакрытые → `status=blocked` (юзер решил остановиться).** |
| активен → на паузе | Юзер сказал «поставь на паузу» | Дедлайны замораживаются, не сдвигаются. В истории — дата паузы. **В `state/tasks.yaml` все записи этой цели → `status=planned` (заморожены, не трогаем до возврата).** |
| на паузе → активен | Юзер сказал «продолжаем» | Дедлайны размораживаются. Если пауза была >14 дней — предложить ревизию плана. **В `state/tasks.yaml` все записи этой цели → восстанавливаются, `deadline` пересчитывается при необходимости.** |
| активен/на паузе → отменён | Юзер сказал «закрываем» / «забили» | Файл **не удаляется**, помечается `cancelled`, остаётся в `plans/` для истории. **В `state/tasks.yaml` все записи этой цели → `status=cancelled` или удаляются, если юзер хочет чистый дашборд.** |

## Анти-паттерны

- ❌ Один план без альтернатив
- ❌ План без research (на глаз)
- ❌ Поверхностные черновики без конкретных занятий по дням
- ❌ Магическое «делай столько-то часов» без логики расчёта
- ❌ Допрос 10+ вопросами при сборе входных
- ❌ Забыть про уже идущие планы и перегрузить пользователя
- ❌ Нечестно обещать нереальное («успеешь, если постараешься»)
- ❌ Игнорировать materials, которые у пользователя уже есть
- ❌ Плодить копии файлов при обновлении
- ❌ «Материнский тон»: «давай, ты сможешь!», «не сдавайся!»
- ❌ Привязка к конкретному пользователю (hardcoded предпочтения, лимиты)
- ❌ Дать пользователю незаметно сьехать с темы на побочные задачи, не обозначив это
- ❌ Запрещать дрейф жёстко (это не тюрьма) — но и не замечать его
- ❌ Самоотключаться после выбора плана (навык остаётся активным до явного «закрываем»)
- ❌ Вкладывать материалы в план дня (видео, ссылки, тексты задач) — это забивает план и ломает его как рабочий инструмент
- ❌ Выдавать материалы проактивно, без запроса
- ❌ Не сообщить правило о материалах при первой выдаче плана
- ❌ Считать «день N» по наитию — только по журналу
- ❌ Выдавать план дня с суммой часов > дневной нагрузки (без пометки)
- ❌ Игнорировать «нет отчёта» за вчера — молча продолжить как будто всё ок
- ❌ Строить месячный план, когда до дедлайна <24 ч (вместо triage)
- ❌ Использовать размытые пороги «>N дней» — только конкретные 1 / 2-3 / >3
- ❌ Принимать вес ≤ 0, дедлайн в прошлом, бюджет = 0 без явного уточнения
- ❌ Говорить «план сохранён», не проверив, что файл записан
- ❌ Использовать слаг с пробелами/спецсимволами или перезаписывать существующий файл без вопроса
- ❌ Писать в `~/.openclaw/workspace/plans/<slug>.md` вместо `users/<user_key>/plan.md` (path conflict с user-profile)
- ❌ Определять текущую фазу по наитию — есть правило в шаге 9
- ❌ После адаптации (шаг 7) выдавать дневной план по старой фазе
- ❌ При multi-goal выдавать 2 отдельных дневных плана (нужен единый сводный)
- ❌ Не зафиксировать drift-событие в логе
- ❌ Удалять файл плана при отмене (legacy в `plans/` остаётся для истории, `users/<key>/plan.md` тоже сохраняется со статусом `cancelled`)
- ❌ Создать план без записи задач в `state/tasks.yaml` (task-tracker не узнает о цели)
- ❌ Забыть `user_key` в записи task (multi-user сломается)
- ❌ Использовать `weight` 1-10 вместо 1-5 (рассогласование с daily-plan)
- ❌ Изменить milestone'ы в файле плана без обновления `state/tasks.yaml` (расходятся)
- ❌ Main сам делает `web_search` / `web_fetch` в шаге 2.10 БЕЗ fallback-логики (если субагента нет — нужно self-research с пометкой)
- ❌ Забыть сохранить копию research в Obsidian (`Для openclaw/research/<goal-slug>.md`)

## Связанные навыки

- `evening-task-triage` — для ежедневного разбора каши в голове
- `task-tracker` — для отслеживания прогресса, дашборда, статистики (← единый источник истины)
- `task-triage` — для декомпозиции и capability-выполнения
- `goal-checkin-notifier` — для Telegram-напоминаний (читает `state/tasks.yaml`)
- `idea-tools` — `op=promote` для превращения топ-идей в задачи
- `cron` — для напоминаний и регулярных check-in
- `material-cache` — делит с этим навыком субагента `research-assistant`; используется для ад-хок запросов материалов («дай разбор задачи X»). Оба пишут в общий `knowledge/<subject>/` — main workspace.
- `skill_workshop` — для обновления этого навыка
